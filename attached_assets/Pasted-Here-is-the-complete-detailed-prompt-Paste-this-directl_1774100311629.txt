### Prompt for Replit AI – Table QR Customer Request System

This is a continuation of the existing restaurant SaaS project. The following modules are already built and working: multi-tenant architecture, role-based access, POS, KDS, tables, menu, orders, and QR-based ordering.

Now build a **complete Table QR Customer Request System** where customers can make all types of requests directly from their table by scanning a QR code — from ordering food to requesting cleaning, calling a waiter, asking for the bill, and more.

***

#### 1. Core concept and flow

```
Customer sits at table
        ↓
Scans QR code on table
        ↓
Opens mobile-friendly web page (no app download, no login)
        ↓
Sees request options + menu
        ↓
Submits any request
        ↓
Request appears INSTANTLY on:
  ├── Waiter's tablet/phone (notification)
  ├── Manager dashboard (alert panel)
  └── Relevant staff screen (cleaning, bar, kitchen)
        ↓
Staff acknowledges + acts on request
        ↓
Customer sees real-time status update
```

***

#### 2. QR code generation per table

- Each table must have a **unique QR code** that encodes:
  ```
  https://[app-domain]/table/[tenant_slug]/[outlet_id]/[table_id]?token=[secure_token]
  ```
- QR code properties:
  - Unique per table, per outlet, per tenant.
  - Contains a **secure token** (UUID or JWT) to prevent spoofing.
  - Token does not expire (permanent for the table) but can be regenerated by manager.
- In Table Management settings:
  - Show QR code per table with options:
    - Download as PNG (for printing and laminating on table).
    - Download as PDF (print-ready with table number, restaurant name, logo).
    - Regenerate QR (invalidates old token).
    - Bulk download all table QR codes as a ZIP.
  - QR code design:
    - Include restaurant logo in center of QR.
    - Table number printed below QR.
    - "Scan to order or request assistance" text.

#### 5. Database schema

```sql
-- Customer requests from QR
CREATE TABLE table_requests (
  id               UUID PRIMARY KEY,
  tenant_id        UUID NOT NULL,
  outlet_id        UUID NOT NULL,
  table_id         UUID NOT NULL,
  table_number     VARCHAR(20),
  request_type     VARCHAR(50) NOT NULL,
    -- 'CALL_WAITER', 'REQUEST_BILL', 'REQUEST_WATER',
    -- 'REQUEST_CLEANING', 'REQUEST_CUTLERY',
    -- 'ORDER_FOOD', 'FEEDBACK', 'COMPLAINT',
    -- 'SPECIAL_REQUEST', 'OTHER'
  request_category VARCHAR(30),
    -- 'ASSISTANCE', 'ORDER', 'FEEDBACK', 'SPECIAL'
  priority         VARCHAR(10) DEFAULT 'medium',
    -- 'high', 'medium', 'low'
  status           VARCHAR(20) DEFAULT 'pending',
    -- 'pending', 'acknowledged', 'in_progress',
    -- 'completed', 'cancelled'
  details          JSONB,
    -- flexible field for any additional data
  customer_note    TEXT,
  assigned_to      UUID,
  assigned_name    VARCHAR(255),
  acknowledged_at  TIMESTAMP,
  completed_at     TIMESTAMP,
  created_at       TIMESTAMP DEFAULT NOW(),
  updated_at       TIMESTAMP DEFAULT NOW()
);

-- QR table tokens
CREATE TABLE table_qr_tokens (
  id           UUID PRIMARY KEY,
  tenant_id    UUID NOT NULL,
  outlet_id    UUID NOT NULL,
  table_id     UUID NOT NULL,
  token        VARCHAR(255) UNIQUE NOT NULL,
  is_active    BOOLEAN DEFAULT true,
  generated_at TIMESTAMP DEFAULT NOW(),
  generated_by UUID
);
```

#### 6. Real-time request routing and notifications

When a customer submits any request:

- **Immediately broadcast** via WebSocket to:
  - All waiters assigned to that table's section.
  - Manager dashboard.
  - Relevant staff based on request type:
    - Cleaning request → cleaning staff screen.
    - Bill request → cashier screen.
    - Food order → KDS (kitchen).
    - Drink order → bar display.

- **In-app notification** appears as:
  ```
  🔔 Table 5 — Request Cleaning
  2 seconds ago | [Acknowledge] [Assign]
  ```

- **Sound alert** on staff screens (different tones per priority):
  - High priority: urgent chime (3 beeps).
  - Medium priority: soft chime (1 beep).
  - Low priority: silent (badge only).

- **Escalation rule**: if a request is not acknowledged within:
  - High priority: 2 minutes → alert manager.
  - Medium priority: 5 minutes → alert manager.
  - Low priority: 10 minutes → alert manager.

#### 13. APIs required

```
GET    /api/qr/:token                      → validate QR token, return table info
POST   /api/table-requests                 → create new request from customer
GET    /api/table-requests/live            → live requests for staff (WebSocket)
PUT    /api/table-requests/:id/acknowledge → staff acknowledges request
PUT    /api/table-requests/:id/complete    → staff completes request
PUT    /api/table-requests/:id/assign      → manager assigns to staff
GET    /api/table-requests/history         → request history with filters
GET    /api/table-requests/analytics       → summary stats for reports
POST   /api/qr/generate/:tableId           → generate/regenerate QR token
GET    /api/qr/download/:tableId           → download QR as PNG/PDF
GET    /api/qr/bulk-download/:outletId     → download all QR codes as ZIP
```
