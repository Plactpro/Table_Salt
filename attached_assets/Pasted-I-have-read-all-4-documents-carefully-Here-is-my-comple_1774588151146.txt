## PR-017 — EU Fiscal Compliance Review

### Assessment: ⚠️ MAJOR CONCERN — Read Carefully

```
The plan is technically well-written.
BUT there is a fundamental problem
that the plan does not address.
```

### The Critical Issue
```
PR-017 builds the TECHNICAL implementation
but misses the LEGAL REALITY:

GERMANY TSE:
  → TSE device must be physically certified
    by BSI (Bundesamt für Sicherheit
    in der Informationstechnik)
  → Your SOFTWARE must be certified to work
    with the certified TSE
  → Swissbit TSE: requires their SDK license
    (not free, requires commercial agreement)
  → EPSON OPOS TSE: same — commercial SDK
  → Without these licenses and certifications:
    → Code runs ✅
    → Legally compliant ❌
    → Fine of €25,000 per violation still applies

ITALY RT:
  → RT device API requires the device to be
    registered with Agenzia delle Entrate
  → Registration is done by the RESTAURANT
    (not by you)
  → Your software must be compatible with
    their specific registered device model
  → Certification: Italian authorities can
    audit your software

FRANCE NF 525:
  → Task correctly says:
    "The actual certification is an operational/
     commercial step outside this codebase"
  → This is honest ✅
  → BUT: NF 525 certification costs
    €5,000–20,000 and takes 3-6 months
  → Without it: French restaurants face
    €7,500 fine per fraudulent receipt
    and tax authority can recompute ALL taxes
    for the last 3 years
```

### My Recommendation on PR-017

```
OPTION A: BUILD AS PLANNED (Current approach)
  → Build the technical infrastructure
  → Mark as "not for production use
    in DE/IT/FR until certification obtained"
  → Use as a foundation for future certification

  PROS: Code is ready when certification comes
  CONS: Restaurant cannot actually use it legally

OPTION B: DEFER ENTIRELY (My recommendation)
  → Remove from current plan
  → Focus on UAE, UK, Singapore, Australia
    (no fiscal certification needed in these)
  → Revisit when you have DE/IT/FR clients
    and budget for certification process

OPTION C: PARTNER APPROACH (Best for business)
  → Partner with a certified fiscal software
    provider in each country
  → White-label their compliance module
  → Your code integrates with their certified layer
  → No certification burden on you
  → Example partners:
    Germany: DATEV, GASTRO-MIS (certified)
    Italy: Epson, Custom, Cassa in Cloud
    France: Lightspeed NF525, L'Addition

MY RECOMMENDATION: Option B now, Option C later

  Do NOT spend Replit AI sessions building
  fiscal compliance that cannot be legally
  used without €20,000+ certification.
  
  The UAE, UK, Singapore, Australia markets
  are larger opportunities AND require
  NO fiscal certification.
  
  Defer PR-017 to Year 2.
```

### Assessment: ✅ EXCELLENT — Best of the 4 Plans

```
WHAT IS CORRECT:
  → API key prefixes ts-live_ and ts-test_
    (Stripe-style — industry standard)
  → Key shown ONCE on creation ← correct security
  → bcrypt hash stored, never plaintext ← correct
  → Scoped API keys ← correct
  → Versioned API (/api/v1) ← critical for future
  → Rate limiting: 60 req/min per key ✅
  → Webhook HMAC-SHA256 signing ✅
  → X-TableSalt-Signature header ✅
  → Exponential backoff: 1min, 5min, 30min,
    2hr, 8hr, 24hr (6 attempts) ✅
  → Fire-and-forget (never blocks request) ✅
  → Webhook delivery log ✅
  → Resend button on failed deliveries ✅
  → QuickBooks + Xero daily export ✅
  → OpenAPI spec for documentation ✅
  → Sandbox using demo tenant ← smart

EXCELLENT DESIGN DECISIONS:
  → "Read-only keys are safe for BI tools"
    ← operationally correct
  → "Keys can be named" ← helpful for multi-system
  → Standard response envelope
    {success, data, error, meta} ← correct
  → Rate limit headers in response ← developer friendly
```

### Minor Gaps
```
GAP 1: Webhook ping on creation
  → "Validates URL is reachable with a test ping"
  → Good ✅

  → And: timeout should be 5 seconds
  → Firewall rules may block unexpected GETs

GAP 2: QuickBooks OAuth complexity
  → QuickBooks uses OAuth 2.0 with refresh tokens
  → Tokens expire and must be refreshed
  → Needs token refresh job in retention scheduler
  → Task 7 mentions OAuth but verify
    token refresh is explicitly handled

GAP 3: Xero scope limitation
