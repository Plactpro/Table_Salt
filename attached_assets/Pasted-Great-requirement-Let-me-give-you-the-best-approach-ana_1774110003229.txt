## Best Approach Analysis

### The Problem
```
Multiple cooking counters → Multiple chefs → Orders arriving continuously
How to assign dishes to the right chef at the right counter efficiently?
```

### 3 Possible Approaches

| Approach | How It Works | Best For | Weakness |
|---|---|---|---|
| **A. Pure Auto-Assign** | System assigns based on roster/workload | Large restaurants, high volume | Chefs lose control |
| **B. Pure Self-Assign** | Chef picks their own tickets | Small restaurants, low volume | Tickets ignored, chaos |
| **C. Hybrid (Best ✅)** | Auto-assign first, chef can override/self-assign | All sizes | Needs smart logic |

### Recommended: Hybrid Intelligent Assignment
```
Order arrives
    ↓
System tries AUTO-ASSIGN first:
  → Check roster (who is scheduled for this shift?)
  → Check counter (which counter handles this dish type?)
  → Check workload (who has fewest active tickets?)
  → Assign to best available chef
    ↓
If no chef available:
  → Ticket goes to UNASSIGNED pool
  → Any chef can SELF-ASSIGN from pool
    ↓
If ticket not picked in X minutes:
  → Escalate to Head Chef / Manager
```

#### 1. Core entities and concepts

```
RESTAURANT
  └── OUTLET
        └── COUNTERS (physical cooking stations)
              └── CHEFS (assigned per shift via roster)

```

- **Counter**: a physical cooking station (Grill, Cold Kitchen, Bar, Fry Station, Bakery, etc.).
- **Chef**: a kitchen staff member assigned to one or more counters per shift.
- **Roster**: shift schedule that defines which chef works at which counter during which time.
- **Ticket**: a KOT (Kitchen Order Ticket) for a specific dish that needs to be prepared.
- **Assignment**: the link between a ticket and a chef/counter.

#### 5. Hybrid intelligent assignment engine

This is the core logic. Implement as a service: `ChefAssignmentService`.

##### Algorithm (runs every time a new ticket arrives):

```typescript
async function assignTicket(ticket: Ticket): Promise<Assignment> {

  // STEP 1: Determine which counter handles this dish
  const counter = await getCounterForMenuItem(ticket.menuItemId);

  // STEP 2: Get all chefs available at this counter right now
  const availableChefs = await getAvailableChefs(
    counter.id,
    ticket.outletId,
    currentShift()
  );

  if (availableChefs.length === 0) {
    // No chef available → send to UNASSIGNED pool
    return assignToPool(ticket, counter);
  }

  // STEP 3: Score each available chef
  const scored = availableChefs.map(chef => ({
    chef,
    score: calculateScore(chef, ticket)
  }));

  // STEP 4: Pick highest scoring chef
  const best = scored.sort((a, b) => b.score - a.score)[0];

  // STEP 5: Assign ticket
  return createAssignment(ticket, best.chef, counter);
}

function calculateScore(chef, ticket): number {
  let score = 100;

  // Factor 1: Workload (fewer active tickets = higher score)
  score -= chef.activeTickets * 20;

  // Factor 2: Roster match (scheduled for this counter = bonus)
  if (chef.isRosteredForCounter(ticket.counterId)) score += 30;

  // Factor 3: Specialization (chef has prepared this dish before)
  if (chef.hasExperienceWith(ticket.menuItemId)) score += 15;

  // Factor 4: Recent performance (faster avg prep time = bonus)
  score += Math.max(0, 20 - chef.avgPrepTimeMinutes);

  return score;
}
```

##### Assignment modes (configurable per outlet):

```
MODE 1: FULL AUTO
  → System always assigns; chef cannot self-assign
  → Best for: high-volume, disciplined kitchens

MODE 2: HYBRID (DEFAULT ✅ RECOMMENDED)
  → System auto-assigns first
  → If unassigned pool has tickets, any on-duty chef can grab them
  → Chef can request reassignment with reason

MODE 3: SELF-ASSIGN ONLY
  → All tickets go to pool
  → Chef picks what they want to prepare
  → Best for: small restaurants

MODE 4: MANUAL (Manager-only)
  → Manager assigns each ticket manually
  → Best for: high-end fine dining with specific pairings
```
