# 🔴 BUG-01 — Tables Page Crashes on First Load (React Error #185)

| Field | Details |
|---|---|
| **Bug ID** | BUG-01 |
| **Title** | `/tables` route crashes with React Error #185 on every first page load |
| **Module** | Tables & Queue Management |
| **Route** | `https://inifinit.com/tables` |
| **Severity** | 🔴 Critical |
| **Priority** | P1 — Must Fix Immediately |
| **Status** | ⚠️ Partial Fix (crashes first, works after Retry) |
| **Affected Roles** | Owner, Manager, Waiter |
| **Found In** | Round 1 (4 PM) |
| **Retest Result** | Round 2 (8 PM): Error message changed but crash persists on first load |

### Steps to Reproduce
1. Log in as any role with Tables access (Owner: `owner / demo123`)
2. Click **"Tables"** in the left sidebar — OR — directly navigate to `https://inifinit.com/tables`
3. Observe the page

### Actual Result
- Page shows error UI: *"Could not load tables — Minified React error #185"*
- All table data, floor plan, and table management features are completely inaccessible
- Page cannot be used in this state

### Expected Result
- Tables page loads directly showing the full floor plan with zone-wise table layout
- Occupancy stats (Available, Occupied, Reserved, Waitlist) displayed correctly

### Workaround
- Click the **"Retry"** button shown in the error UI — page loads successfully on second attempt
- This is a temporary workaround and is NOT acceptable in production

### Root Cause (Suspected)
- React Error #185 = "Invalid hook call" — a React hook is being called conditionally or outside a component
- Likely a hook (e.g. `useEffect`, `useFetch`, `useContext`) that runs during initial mount with undefined/null data, causing the component tree to fail
- May be caused by the `outlet_id` or `restaurant_id` context being null during first render before the auth context hydrates fully
- On retry, the context is already hydrated so the hook call succeeds

### Screenshots / Evidence
- Round 1: Error showed generic "Something went wrong"
- Round 2: Error upgraded to "Minified React error #185" with Retry button (partial improvement)

### Impact
- **Dine-in orders are blocked** until tables load (BUG-02 and BUG-03 are cascading from this)
- Every user who navigates to Tables sees a crash on first visit — affects all shifts daily
- Creates a false impression of a broken product to new users and restaurant owners

### Fix Recommendation
```javascript
// In the Tables component, ensure the hook is called unconditionally
// and that empty/null data is handled gracefully before render:

const { data: tables, loading, error } = useFetchTables(outletId);

// Add null guard before rendering:
if (loading) return <LoadingSpinner />;
if (error) return <EmptyState message="Could not load tables" retry={refetch} />;
if (!tables || tables.length === 0) return <EmptyState message="No tables configured yet" cta="Add Table" />;
```

# 🔴 BUG-03 — "Send to Kitchen" Triggers `about:blank#blocked` (Popup Blocked)

| Field | Details |
|---|---|
| **Bug ID** | BUG-03 |
| **Title** | Clicking "Send to Kitchen" in POS navigates to `about:blank#blocked` due to blocked popup |
| **Module** | POS — Order Submission / KOT Printing |
| **Route** | `https://inifinit.com/pos` |
| **Severity** | 🔴 Critical |
| **Priority** | P1 — Must Fix Immediately |
| **Status** | ❌ Open — Not Fixed |
| **Affected Roles** | Waiter, Manager, Owner |
| **Found In** | Round 1 (4 PM) |
| **Retest Result** | Round 2 (8 PM): Still occurring. However, order IS saved — only the KOT print popup is blocked |

### Steps to Reproduce
1. Log in as Waiter (`waiter / demo123`)
2. Navigate to POS (`/pos`)
3. Select **Dine-in** → Select a table (e.g. Table 2 — Main Hall)
4. Add any menu items (e.g. Calamari Fritti x2, Penne Arrabbiata x2)
5. Scroll down in the right panel
6. Click the **"Send to Kitchen"** button (teal button, bottom right)

### Actual Result
- Browser navigates away to `about:blank#blocked`
- The active page is replaced with a blank page
- User must manually navigate back to `/pos`
- **Note (positive finding):** The order IS successfully placed in the database — visible in KDS and Online Orders — only the KOT print popup is blocked

### Expected Result
- Order sent to Kitchen Display System (KDS) successfully
- KOT (Kitchen Order Ticket) printed or displayed in-app
- POS resets to fresh order / shows confirmation toast
- User stays on `/pos` page throughout

### Root Cause (Confirmed)
```javascript
// PROBLEM: window.open() called inside async callback — browser blocks it
const handleSendToKitchen = async () => {
  await createOrder(orderData);           // async — loses user gesture context
  window.open(kotPrintUrl, '_blank');     // BLOCKED — not in sync user gesture
};
```
Browser popup blockers block any `window.open()` that is called inside an `async/await` chain or Promise `.then()` because it is no longer considered a direct result of a user gesture.

### Impact
- **Every dine-in and takeaway "Send to Kitchen" action breaks the UI** — staff must navigate back manually every single order
- Kitchen may not receive the printed KOT ticket (even though order is in KDS)
- Extremely disruptive to daily restaurant operations — affects every waiter on every shift

### Fix Recommendation
**Option A (Recommended) — In-App Print using `react-to-print`:**
```javascript
import { useReactToPrint } from 'react-to-print';

const kotRef = useRef();
const handlePrint = useReactToPrint({ content: () => kotRef.current });

const handleSendToKitchen = async () => {
  const order = await createOrder(orderData); // async
  setKotData(order);                          // set KOT data
  handlePrint();                              // print from hidden ref — no popup
};

// In JSX: Hidden KOT template
<div style={{ display: 'none' }}>
  <KOTTemplate ref={kotRef} data={kotData} />
</div>
```

**Option B — Open popup synchronously BEFORE async call:**
```javascript
const handleSendToKitchen = async () => {
  const printWindow = window.open('', '_blank'); // SYNC — opens on click gesture
  const order = await createOrder(orderData);    // ASYNC — after window opened
  printWindow.document.write(generateKOTHtml(order));
  printWindow.print();
  printWindow.close();
};
```

# 🔴 BUG-06 — KDS "Confirm & Start Cooking" Triggers `about:blank#blocked`

| Field | Details |
|---|---|
| **Bug ID** | BUG-06 |
| **Title** | KDS "Confirm & Start Cooking" button navigates to `about:blank#blocked` |
| **Module** | Kitchen Display System (KDS) |
| **Route** | `https://inifinit.com/` (Kitchen role dashboard) |
| **Severity** | 🔴 Critical |
| **Priority** | P1 — Must Fix Immediately |
| **Status** | ❌ Open — Not Fixed |
| **Affected Roles** | Kitchen Staff |
| **Found In** | Round 1 (4 PM) |
| **Retest Result** | Round 2 (8 PM): Still occurring |

### Steps to Reproduce
1. Log in as Kitchen Staff (`kitchen / demo123`)
2. KDS loads with NEW / COOKING / READY columns
3. Find a ticket in the **NEW** column
4. Click **"Start All"** button on the ticket
5. "Recipe Check — Start Cooking" modal appears with ingredient validation results
6. Click **"Confirm & Start Cooking"** button

### Actual Result
- Browser navigates to `about:blank#blocked`
- KDS is gone from view — kitchen staff must manually go back
- It is **unknown whether the ticket was moved to COOKING** status or not (cannot verify due to navigation away)

### Expected Result
- Ticket moves from NEW → COOKING column in real-time
- KOT print triggers (in-app or via connected kitchen printer)
- Success toast: "Order #XXXX is now Cooking"
- Kitchen staff remains on KDS page

### Root Cause (Confirmed)
- Same root cause as BUG-03 — `window.open()` called inside an async function chain, blocked by browser popup blocker
- The Recipe Check modal button's click handler fires an async API call to mark the ticket as cooking, then attempts `window.open(kotUrl)` — blocked because it's not a synchronous user gesture anymore

### Impact
- **Kitchen staff cannot confirm cooking status for any order** — this breaks the entire order lifecycle
- KDS is the core of kitchen operations; if staff cannot confirm tickets, COOKING and READY statuses are unreliable
- Ticket timers become inaccurate
- Orders may be forgotten with no status tracking

### Fix Recommendation
Same as BUG-03. Replace `window.open()` with in-app `react-to-print` OR open the window synchronously before the async call. Since BUG-03 and BUG-06 share the same root cause, a single shared fix in the print utility function will resolve both.
