### Prompt for Replit AI – Full SaaS Restaurant Management System

You are an expert SaaS architect and full‑stack engineer.
Design and implement a **multi‑tenant Restaurant Management System (RMS)** as a SaaS product, targeting **at least 2,000 active restaurant tenants**. The system must cover **end‑to‑end restaurant operations** with a **strong, scalable architecture**, **role‑wise dashboards**, **recipe to inventory linking**, **combo offers with % savings**, and a **30‑day full free trial onboarding flow**.

***

#### 1. Architecture and tech stack

- Build a **cloud‑ready, production‑grade web app**.
- Stack:
  - Frontend: React + TypeScript with Material UI or Tailwind + Headless UI.
  - Backend: Node.js + TypeScript using Express or NestJS with clean layered architecture (routes/controllers → services → repositories).
  - Database: PostgreSQL with migrations, seed data, and Prisma ORM.
  - Auth: JWT + refresh tokens; optional Google OAuth.
- **Multi‑tenant SaaS design**:
  - Each restaurant/brand is a **tenant**.
  - Maintain a `tenants` table and include `tenant_id` on all tenant‑scoped tables.
  - Enforce tenant isolation at API middleware and DB query layers.
- Non‑functional requirements:
  - Support ~2,000 tenants with multiple users each.
  - Stateless backend for horizontal scaling.
  - Rate limiting, input validation, structured logging.
  - Health‑check endpoint for monitoring.

***

#### 2. User roles and permissions

Implement **role‑based access control** with permissions per route and UI:

- Tenant Owner / Admin
- Restaurant Manager
- Waiter / Cashier
- Kitchen / Bar Staff
- Accountant / Finance
- Customer (optional QR/portal)

Every API and page must check both **role** and **tenant_id** before granting access. Navigation menu items are shown or hidden based on role.

#### 3. Tenant onboarding and 30‑day full free trial

Every new restaurant that signs up gets a **30‑day full free trial** with no credit card required at registration.

**Sign‑up flow:**
- Collect: name, work email, password, restaurant name, country.
- Create `user` + `tenant` + `subscription` records in one atomic transaction.
- Immediately log the user in and redirect to an **onboarding wizard** (not a blank dashboard).

**Onboarding wizard (5 steps):**
1. Restaurant basics: logo upload, address, timezone, currency.
2. Outlets and tables: create at least one outlet and a few tables.
3. Menu basics: import a starter template or create top 5 items.
4. Invite team members (optional, skippable).
5. Choose plan: show pricing table for reference (payment not required yet during trial).

**Trial rules:**
- Trial starts on registration date; ends exactly 30 days later.
- During trial: full access to all features identical to the paid Pro plan.
- No payment collected at sign‑up.

**Subscription state machine:**
```
REGISTERED
    ↓
TRIALING (30 days, full access)
    ↓
PENDING_PAYMENT → payment success → ACTIVE_PAID
    ↓ (no payment after trial ends)
TRIAL_EXPIRED (read‑only, data preserved, 15‑day grace)
    ↓ (no payment after grace period)
SUSPENDED (data locked)
```

**Database – subscriptions table:**
```sql
CREATE TABLE subscriptions (
  id              UUID PRIMARY KEY,
  tenant_id       UUID NOT NULL,
  plan_id         UUID,
  status          VARCHAR(30) DEFAULT 'trialing',
  trial_start     DATE NOT NULL,
  trial_end       DATE NOT NULL,
  billing_start   DATE,
  billing_end     DATE,
  payment_method  VARCHAR(50),
  created_at      TIMESTAMP DEFAULT NOW(),
  updated_at      TIMESTAMP DEFAULT NOW()
);
```

**Trial countdown and nudge behavior:**
- Show trial progress bar in top header across all pages:
  - Day 1–20: Green, "X days remaining in your trial."
  - Day 21–27: Amber, "Only X days left – upgrade to keep access."
  - Day 28–30: Red with pulsing animation, "Your trial expires in X days."
- Automated reminder emails at:
  - Day 25: "5 days left – here is what you have built so far."
  - Day 29: "Only 1 day left – upgrade now to avoid interruption."
  - Day 30: "Your trial has ended – add payment to continue."

**On trial expiry (no payment):**
- Lock all write operations; allow read‑only access for 15‑day grace period.
- Show full‑screen upgrade paywall with plan options and payment form.
- After 15‑day grace with no payment: move to `SUSPENDED`.

**On upgrade (during or after trial):**
- Collect payment via Stripe / Razorpay integration placeholder.
- On success: set `status = active_paid`, `billing_start = today`.
- Remove trial bar, show "Plan Activated" success toast.
- All trial data preserved completely; no re‑setup required.

**Super admin (platform owner) view:**
- All tenants with trial status, days remaining, and conversion rate.
- Manually extend trial for specific tenants.
- Trial conversion funnel: registered → activated → trial active → paid → churned.

#### 4. Role‑wise multiple dashboards

Implement **separate dashboard layouts** for each role using a shared widget system:

- Common approach:
  - Reusable dashboard widgets (cards, charts, tables).
  - Each role's dashboard is a configured set of widgets.
  - On login, route to the appropriate dashboard; protect routes by role.

- **Tenant Owner / Admin Dashboard**
  - Metrics: total revenue across outlets, outlet‑wise sales comparison, total orders, avg order value, plan usage, staff count, low‑stock summary, recent activity log.
  - Actions: manage outlets, manage subscription, configure settings, access global reports.

- **Restaurant Manager Dashboard**
  - Scope: assigned outlet(s) only.
  - Metrics: today's sales, orders in progress, table occupancy, upcoming reservations, staff on duty, low‑stock alerts, top 5 menu items.
  - Quick actions: POS, tables, reservations, staff, inventory.

- **Waiter / Cashier Dashboard**
  - My open tables, my open orders, "New Order" button, KOT history, shift summary.
  - No financial or configuration widgets.

- **Kitchen / Bar Staff Dashboard (KDS Home)**
  - Order ticket queue (new, in‑progress, ready).
  - Visual/sound alert for new tickets.
  - Stats: tickets today, average prep time.

- **Accountant / Finance Dashboard**
  - Financial widgets only: sales by date range, taxes collected, discounts/voids, export shortcuts.

- UX:
  - Use distinct icon sets and color themes per role while keeping a consistent design system.
  - Subtle entrance animations for widgets on load; hover effects on interactive cards.

#### 5. Menu management with combo offers and % savings

##### 5.1 Categories and items (standard CRUD)

- Category: name, display order, image/color, active flag, multi‑outlet availability.
- Item: name, description, image, base price, tax group, veg/non‑veg, spicy level, tags, availability flag.
- Modifiers/add‑ons with price adjustments.
- Time‑based menus (breakfast/lunch/dinner) and outlet‑specific pricing.
- Link each item to a recipe for auto stock deduction (see Section 6).

##### 5.2 Combo offers with % savings

- Dedicated **"Create Combo Offer"** workflow in Menu section.
- Combo builder:
  - Select component items (main item + side items + optional add‑ons).
  - Set combo name and combo price.
  - System auto‑calculates individual items total and live % savings:
    ```
    savings% = ((individual_total - combo_price) / individual_total) × 100
    ```
- Validation: combo price must be less than individual total (min 5% savings enforced).
- Display in POS and customer portal:
  - Combo price (large, bold)
  - Individual value with strikethrough
  - "SAVE X%" green badge
- Order processing:
  - Combo added as a single line item in the cart.
  - Internally expanded to component items for inventory deduction and KOT.
  - Receipt shows: "Combo Name – ₹X (Save ₹Y)"
- Combo management:
  - Set validity dates and time‑slot availability.
  - Outlet‑specific availability.
  - Auto‑deactivate expired combos.
  - Combo analytics: views, selections, revenue contribution.

**Combo database table:**
```sql
CREATE TABLE combo_offers (
  id                  UUID PRIMARY KEY,
  tenant_id           UUID NOT NULL,
  name                VARCHAR(255) NOT NULL,
  combo_price         DECIMAL(10,2) NOT NULL,
  individual_total    DECIMAL(10,2) NOT NULL,
  savings_percentage  DECIMAL(5,2) NOT NULL,
  components          JSONB NOT NULL,
  validity_start      DATE,
  validity_end        DATE,
  time_slots          JSONB,
  outlet_ids          JSONB,
  is_active           BOOLEAN DEFAULT true,
  created_at          TIMESTAMP DEFAULT NOW()
);
```

#### 6. Recipe to inventory linking

##### 6.1 Ingredient / inventory model

- Fields: `name`, `sku`, `category`, `storage_unit`, `purchase_unit`, `conversion_factor`, `current_stock`, `par_level`, `reorder_level`, `last_purchase_price`, `average_cost`.
- Use a **base unit** internally (grams/ml) with conversion factors so recipes can use user‑friendly units.

##### 6.2 Recipe model

- Recipe: `name`, `description`, `yield_quantity`, `yield_unit`, `waste_percentage`.
- RecipeComponent lines: `ingredient_id`, `quantity`, `unit`, `notes`.
- Auto‑calculated:
  - `line_cost = converted_quantity × ingredient_average_cost`
  - `recipe_total_cost = sum(all line costs) × (1 + waste_percentage)`
  - `cost_per_portion = recipe_total_cost ÷ yield_quantity`

##### 6.3 Menu item ↔ recipe linking

- Attach one primary recipe to each menu item.
- Support portion multipliers for sizes (Small=1.0, Regular=1.5, Large=2.0).
- Menu item page shows: recipe cost per portion, selling price, food cost %.

##### 6.4 Auto stock deduction on sales

- Trigger on order status → "Paid" / "Completed".
- For each order line with a linked recipe:
  ```
  required_qty = component.quantity × (portions_sold / yield_quantity)
  Convert to base unit → deduct from ingredient.current_stock
  ```
- Log every deduction as an `InventoryTransaction` with `type = RECIPE_CONSUMPTION` and reference to `order_id`, `menu_item_id`, `recipe_id`.
- On void/refund/cancel: reverse with `type = RECIPE_REVERSAL`.
- Wrap all deductions in a DB transaction for consistency.

##### 6.5 Recipe editor UI

- Left: recipe header (name, yield, waste %).
- Right: ingredient lines table with live cost calculation per line and running total.
- Bottom panel: total cost, cost per portion, food cost % vs selling price.
- Unsaved changes warning on navigation.

##### 6.6 Recipe and inventory reports

- **Theoretical Usage Report**: expected ingredient usage vs actual stock for a date range.
- **Menu Profitability Report**: cost per portion, selling price, food cost %, contribution margin, quantity sold.
- Export CSV / PDF; filter by category, outlet, and date range.
