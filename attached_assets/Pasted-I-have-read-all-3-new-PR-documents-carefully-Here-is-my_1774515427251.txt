## PR-010 — Data Volume & Retention Assessment

### What Is Covered ✅
```
✅ Audit trail archival to separate table
   (12-month rolling, never deleted)
✅ Archive queryable via admin endpoint
✅ Nightly job at 02:00 UTC
✅ Auto-acknowledge low priority (24h)
✅ Auto-acknowledge medium priority (48h)
✅ Critical/high: NEVER auto-acknowledged
✅ auto_acknowledged column to distinguish
✅ Critical/Info tabs in notification panel
✅ Bell badge: red critical + grey info
✅ Max 99+ display cap
✅ Clear all acknowledged button
✅ Report cache cleanup (48h + last 5 per type)
✅ Stale generating/failed entries (1h cleanup)
```

### Gaps and Observations
```
GAP 1: Nightly job timing conflict
  → Job runs "once on process start,
    then every 24 hours at 02:00 UTC"
  → If server restarts at 01:55 UTC:
    → Job runs immediately at 01:55
    → Then schedules next run 24h later
    → = 01:55 UTC next day
    → Drifts over time away from 02:00
  → MINOR: use a proper daily cron check
    instead of "process start + 24h interval"
  → Check: is current time between 02:00-02:05?
    If yes and not run today: run now
    This is more reliable

GAP 2: Archive query performance
  → Task 1 adds index on tenant_id, created_at
  → But archive table will grow indefinitely
  → After 3 years: archive has 1.3B rows
    (100 restaurants × 36K/month × 36 months)
  → Index helps but still slow at this scale
  → RECOMMENDATION FOR FUTURE:
    Add year-based partitioning to archive table
    Not needed now but plan for it
  → Mark as PR-013 future item

GAP 3: Notification cleanup endpoint
  → Task 5: hard delete acknowledged low/info
  → These are "informational and not compliance"
  → CORRECT — this is safe to hard delete
  → But verify: do any info notifications
    appear in compliance report?
  → If yes: soft delete instead of hard delete

THESE ARE MINOR — PR APPROVED AS-IS
```

### Gaps and Observations
```

  → "Allows 1 test request, closes on success"
  → What if the test request also fails?
  → Should transition back to OPEN immediately
  → Then retry after another 30 seconds
  → Verify this is in the implementation:
    HALF_OPEN → request fails → OPEN (reset timer)
    HALF_OPEN → request succeeds → CLOSED

GAP 2: Circuit breaker scope
  → Applied to: orders, billing, kitchen, reports
  → NOT applied to: auth, health, menu GET
  → QUESTION: what about inventory endpoints?
  → Stock adjustment during peak could be slow
  → RECOMMENDATION:
    Also apply to: inventory mutations
    Not inventory reads (menu GET pattern)

  → "Record manual payment with method=manual_pending"
  → This is correct for the moment
  → BUT: how does manager know to follow up?
  → Need: daily report of manual_pending payments
    "3 payments recorded as manual — gateway down
     Please process these: [list]"
  → Add to PR-010 or as addition here
  → LOW PRIORITY but operationally important

GAP 4: Webhook monitoring — false positives
  → "If last_webhook_at > 2 hours during open hours"
  → What about slow days? (Monday lunch, etc.)
  → Restaurant may genuinely have no orders
    for 2+ hours on a slow day
  → Alert fires unnecessarily
  → Manager ignores alerts ("false alarm again")
  → RECOMMENDATION:
    Only alert if outlet is in "busy period"
    (e.g., weekend, or past hour had 5+ orders)
    → Simple threshold: alert only if
      previous_hour_order_count > 0
    → This prevents false positives on slow days
```

## PR-012 — QR Page Optimization Assessment

### What Is Covered ✅
```

✅ Excludes admin components, charts, etc.

✅ Session expires 30 min after bill paid
✅ Cleanup job for expired sessions
✅ Error states: invalid token, kitchen closed,
   table unavailable, menu load failed ← all covered
✅ "Call for help" button on all error states
✅ Lighthouse target: > 85 score
✅ FCP target: < 1.5 seconds
✅ TTI target: < 3 seconds
```

### Gaps and Observations
```
GAP 1: Bundle size verification
  → Task 1: "Verify with npm run build and
    check generated chunk — target 300KB gzipped"
  → This is correct approach
  → BUT: needs to be actually measured and confirmed
  → If shadcn/ui components are included
    even the "subset" may exceed 300KB
  → RECOMMENDATION:
    After build: document actual bundle size
    in PR notes for future reference
  → If over 300KB: identify which component
    is causing bloat and replace with
    lighter alternative

GAP 2: QR session — concurrent join timing
  → Customer A and Customer B both scan at same time
  → Both call GET /api/qr/[token]/session
  → Both see: sessionExists = false
  → Both call POST /api/qr/[token]/session/start
  → Two sessions created for same table
  → Race condition
  → SOLUTION:
    Use INSERT ... ON CONFLICT DO NOTHING
    on table_id + is_active = true unique constraint
    Second insert fails silently
    Return existing session to second caller

GAP 3: Join existing — cart UX
  → Customer joins existing order
  → They see the combined cart
  → But they can only see items THEY added
  → Or can they see ALL items?
  → CLARIFY:
    → Show all items (full transparency) ← better
    → OR: show only own items ← privacy option
  → Recommendation: show all items with
    "Added by others" label
  → Makes it easier to avoid duplicates
    ("don't order Coke — someone already did")

GAP 4: Error state — "Call for help" button
  → Triggers a waiter request
  → But QR token may be invalid
  → How does system know which table to alert?
  → If token is invalid: cannot identify table
  → SOLUTION:
    → Invalid token: show restaurant phone number
    → Valid token but other error: use table from token
    → This distinction matters for the implementation

GAP 5: Kitchen closed — next open time
  → "We reopen at [next open time from outlet hours]"
  → Outlet hours must be accessible in QR bundle
  → Verify: outlet hours are part of the QR
    token validation response
  → If not: add to GET /api/qr/[token] response

THESE ARE MINOR — PR APPROVED WITH NOTES
```
