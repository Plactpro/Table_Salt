### Prompt for Replit AI – Recipe to Inventory Linking (Auto Stock & Costing)

You are an expert full‑stack engineer working on a multi‑tenant restaurant management SaaS.  
Implement a **Recipe Management module that is tightly linked to Inventory**, so every sale of a menu item automatically consumes the correct ingredients from stock and calculates food cost and margins.

#### 1. Context and architecture

- Stack already in use: React + TypeScript frontend, Node.js + TypeScript backend (Express or NestJS), PostgreSQL + Prisma, multi‑tenant with `tenant_id` on all data.  
- This task is to add/extend:
  - Inventory model (ingredients)  
  - Recipe model  
  - Menu ↔ Recipe linking  
  - Auto inventory deduction and recipe costing

Design clean APIs, database schema, and UI components to support this end‑to‑end.

#### 2. Inventory / Ingredient model

Create or extend an **Ingredient** (inventory item) entity:

- Fields (per tenant):
  - `id`, `tenant_id`
  - `name`, `sku`, `category` (dry, chilled, beverage, etc.)
  - `storage_unit` (kg, litre, piece, pack)
  - `purchase_unit` and `conversion_factor` (e.g., 1 case = 12 bottles, 1 kg = 1000 g)
  - `current_stock` (in base unit)
  - `par_level`, `reorder_level`
  - `last_purchase_price`, `average_cost`
- Ensure all inventory logic uses a **base unit** internally (e.g., grams or ml) so recipes can use user‑friendly units with conversions.

#### 3. Recipe model and schema

Create a **Recipe** entity and **RecipeComponent** lines:

- Recipe:
  - `id`, `tenant_id`
  - `name`
  - `description`
  - `yield_quantity` (number of portions / batch size)
  - `yield_unit` (portion, litre, tray, etc.)
  - `waste_percentage` (optional, default 0)
  - `created_by`, `created_at`, `updated_at`
- RecipeComponent:
  - `id`, `tenant_id`
  - `recipe_id` (FK)
  - `ingredient_id` (FK to inventory item)
  - `quantity` (amount used in the recipe)
  - `unit` (g, ml, pcs, etc.)
  - `notes` (e.g., chopped, diced)
- Back‑end calculations:
  - Convert component `quantity` to ingredient base unit using conversion factors.
  - Compute:
    - `line_cost = converted_quantity × ingredient_average_cost`
    - `recipe_total_cost = sum(line_cost)` (including optional waste factor)
    - `cost_per_portion = recipe_total_cost ÷ yield_quantity`

#### 4. Linking recipes to menu items

Extend **Menu Item** model to link with recipes:

- Fields:
  - `recipe_id` (nullable FK)
  - Optional `portion_multiplier` for each size (Small, Regular, Large).  
- Rules:
  - One **primary recipe** per menu item.  
  - A menu item without a recipe does **not** affect inventory.  
  - For items with multiple sizes, store size/portion definitions with multipliers (e.g., Small=1.0, Large=1.5 portions of the base recipe).

UI in Menu Management:

- “Recipe” tab for each menu item:
  - Select existing recipe or create new.  
  - Show calculated `cost_per_portion`, selling price, and food cost % (`cost_per_portion / selling_price * 100`).

#### 5. Auto stock deduction on sales

In the POS / order flow, implement **automatic inventory deduction** based on recipes:

- Trigger on:
  - Order completion or when order status changes to “Paid” / “Completed”.
- Algorithm for each order line:
  - If menu item has `recipe_id`:
    - Read recipe components and `yield_quantity`.
    - Determine **portions sold**:
      - `portions_sold = line_quantity × portion_multiplier` (if size).  
    - For each component:
      - `required_quantity_recipe_unit = component.quantity × (portions_sold / yield_quantity)`  
      - Convert to ingredient base unit.  
      - Subtract from `ingredient.current_stock`.
- Persist each deduction as an **InventoryTransaction** row:
  - `id`, `tenant_id`
  - `ingredient_id`
  - `type = 'RECIPE_CONSUMPTION'`
  - `quantity_change` (negative)
  - `order_id`, `menu_item_id`, `recipe_id`
  - `created_at`
- On order **void / refund / cancel**, reverse consumption:
  - Create `type = 'RECIPE_REVERSAL'` transaction with positive quantities, updating stock back.

#### 6. Recipe costing and profitability

Expose recipe costing in UI and APIs:

- For each menu item:
  - Display:
    - `recipe_cost_per_portion`
    - `selling_price`
    - `food_cost_percentage`
- Create a “Menu Profitability” report:
  - Columns: menu item, recipe cost, selling price, food cost %, contribution margin, total qty sold.
  - Sort and filter by outlet, category, and date range.

#### 7. Recipe editor UX (frontend)

Build a **Recipe Editor** page:

- Left side: Recipe header
  - Name, yield quantity, yield unit, waste %.
- Right side: Ingredient lines table
  - Ingredient selector (search inventory), quantity, unit, notes column.
  - Show line cost and running total cost (live recalculation).
- Bottom panel:
  - Total recipe cost
  - Cost per portion
  - Optional suggested selling price based on target food cost % (e.g., 30%).  
- UX details:
  - Add ingredient row, duplicate row, delete row.
  - Validation for missing ingredients, zero/negative quantities.
  - Unsaved changes warning on navigation.

#### 8. Inventory & recipe reports

Implement additional reports:

- **Theoretical Usage Report**
  - For a given date range and outlet:
    - For each ingredient, compute total expected usage based on sales × recipes.
    - Show alongside actual stock movements / closing stock.
    - Highlight variance (over/under usage) in quantity and value.
- **Recipe List Report**
  - List all recipes with:
    - Linked menu items
    - Recipe total cost, cost per portion, and cost % vs selling price.
- Support export to CSV / PDF and filtering by category, outlet, and active/inactive status.

#### 9. API endpoints (high level)

Design REST endpoints such as:

- `POST /api/recipes` – create recipe with components.  
- `GET /api/recipes` – list recipes (with pagination & search).  
- `GET /api/recipes/:id` – get recipe + components + calculated costs.  
- `PUT /api/recipes/:id` – update recipe and components.  
- `DELETE /api/recipes/:id` – soft delete (if not in use).  
- `POST /api/menu-items/:id/link-recipe` – link/unlink recipe to menu item.  
- Hook in order completion: service function `applyRecipeConsumption(orderId)` that performs all stock deductions and transactions atomically.

#### 10. Validation, performance, and multi‑tenant safety

- Ensure every recipe, ingredient, and transaction is scoped by `tenant_id`.  
- All queries must filter by `tenant_id` to prevent data leakage.  
- Use DB indexes on `tenant_id`, `ingredient_id`, `recipe_id`, and `order_id` in transactions for performance.  
- Wrap consumption logic in DB transactions to keep data consistent even on errors.

Focus on clean, maintainable code and clear separation of concerns (controllers → services → repositories).  
After implementing, seed demo data with a few recipes (e.g., Burger, Pizza, Coffee) and associated ingredients so the feature can be demonstrated end‑to‑end: create order → see inventory decrease → view recipe cost and usage reports.
