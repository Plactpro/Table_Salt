### SECTION 1 — Offline and Network Resilience

```
1.1 OFFLINE MODE
  ☐ IndexedDB local storage for menu items
  ☐ IndexedDB local storage for open orders
  ☐ Offline order creation (queued locally)
  ☐ Offline KOT display (last known state)
  ☐ Sync queue: auto-sync when back online
  ☐ Conflict resolution on sync
      (server wins for stock/payment data)
  ☐ "Offline Mode" banner shown clearly
  ☐ Offline mode indicator on every screen
  ☐ Operations that work offline:
      → View menu ✓
      → Create order ✓
      → View open orders ✓
      → Print (local network printer) ✓
  ☐ Operations that need internet:
      → Payment processing ✓
      → Real-time KDS sync ✓
      → Reports ✓

1.2 WEBSOCKET RESILIENCE
  ☐ Auto-reconnect on disconnect
      (retry every 3 sec, max 10 attempts)
  ☐ Exponential backoff on reconnect
  ☐ Connection status indicator on ALL screens:
      🟢 Live | 🟡 Reconnecting | 🔴 Offline
  ☐ Queue events during disconnect
  ☐ Replay missed events on reconnect
  ☐ Heartbeat ping every 30 seconds
  ☐ Server-side connection cleanup
      (remove dead connections)
  ☐ Prevent duplicate connections per device

1.3 API REQUEST RESILIENCE
  ☐ Request timeout: 8 seconds max
  ☐ Retry on network failure: 3 attempts
  ☐ Exponential backoff on retry
  ☐ Loading spinner on all API calls
  ☐ Error message on timeout:
      "Request timed out — tap to retry"
  ☐ Idempotency keys on all POST requests
      (prevent duplicate orders/payments)
  ☐ Button disabled immediately after tap
      (prevent double-tap submission)
  ☐ Re-enable button only on response

1.4 DUPLICATE PREVENTION
  ☐ Idempotency key: UUID on every order
  ☐ Server checks: same key = same response
  ☐ Duplicate detection:
      same order + same items + within 10 sec
      → reject second request
  ☐ UI: disable all submit buttons
      immediately on first tap
  ☐ Duplicate order alert to manager
```

### SECTION 4 — Performance and Scalability

```

  ☐ Indexes on ALL foreign keys

      created_at, order_date, staff_id

      max 50 records per page

  ☐ Connection pool: min 5, max 20 connections
  ☐ Database indexes verified and tested
  ☐ EXPLAIN ANALYZE on critical queries

4.2 API PERFORMANCE

  ☐ Heavy reports: background job
      not synchronous API call
  ☐ Report ready notification
      (in-app + email when done)
  ☐ API rate limiting per tenant
  ☐ CDN for static assets (images, JS, CSS)

4.3 FRONTEND PERFORMANCE
  ☐ Code splitting: load only what is needed
  ☐ Lazy loading for non-critical pages
  ☐ Image optimization and compression
  ☐ Menu item images: max 200KB
  ☐ Skeleton loading on all data fetches
  ☐ Virtual scrolling for long lists (50+ items)
  ☐ Debounce on search inputs (300ms)
  ☐ Memoization on expensive renders

4.4 MEMORY MANAGEMENT
  ☐ WebSocket cleanup on component unmount
  ☐ Timer cleanup (clearInterval/clearTimeout)
  ☐ Event listener cleanup on unmount
  ☐ Completed orders removed from memory
      (keep only last 2 hours in state)
  ☐ KDS auto-refresh every 4 hours
      (background refresh, no disruption)
  ☐ Memory usage monitoring
  ☐ Auto-reload on memory threshold exceeded

```

  ☐ Refresh token rotation
  ☐ Refresh token revocation on logout
  ☐ Concurrent session detection:
      "This account logged in on another device.

  ☐ Session invalidation on password change
  ☐ Brute force protection:
      5 failed attempts → 15 min lockout
  ☐ CAPTCHA after 3 failed attempts
  ☐ Suspicious login alert:
      new device, new location
  ☐ 2FA/TOTP for owner and manager roles
  ☐ Quick PIN login for kitchen staff
      (4-digit PIN, no full password needed)

5.2 AUTHORIZATION
  ☐ Every API route checks tenant_id
  ☐ Cross-tenant access returns 403 always
  ☐ Role checked on every sensitive operation
  ☐ Row-level security in PostgreSQL
      (second layer beyond app checks)
  ☐ Cannot escalate own role
  ☐ Super admin actions require
      additional confirmation
  ☐ Read-only mode for impersonation
      (already built — verify working)

5.3 DATA SECURITY

  ☐ HTTPS enforced everywhere (no HTTP)
  ☐ HSTS headers configured
  ☐ CSP (Content Security Policy) headers
  ☐ CORS: only allowed origins
  ☐ API keys in environment variables
      (never in code or client)
  ☐ Audit log on ALL sensitive actions:
      login, logout, delete, payment, refund,
      settings change, role change, export

5.4 PAYMENT SECURITY
  ☐ Card numbers NEVER stored
  ☐ Card numbers NEVER logged
  ☐ PCI DSS SAQ-A documented
  ☐ Payment gateway webhook signature verified
  ☐ Refund requires manager approval
  ☐ Void requires manager approval
  ☐ Payment amount server-validated
      (cannot be modified by client)
  ☐ Double-charge prevention:
      check payment status before processing
