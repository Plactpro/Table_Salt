***

Build fully functional enhancements for the **Table Salt** restaurant management application in three domains:

1. **Self‑Service Channels (Kiosks + Guest Web/QR)**  
2. **Omnichannel Cloud Architecture**  
3. **Security, Governance, and Controls**

Assume the core app already has: multi‑outlet POS, KDS, menu/recipes, CRM, online orders hub, inventory, loyalty, billing, settings (time zone, tax, currency), and cleaning/audit modules. All UI and labels must be **English only**. Use modular HTML/CSS/JavaScript (or TypeScript) and keep code clean and extensible.

## 1. Self‑Service Channels (Kiosks + Guest Web/QR)

### 1.1 Self‑Ordering Kiosk Mode

Create a **full‑screen kiosk interface** optimised for touch devices in QSR/food‑court environments:

- Entry flow:
  - Language selector (for now only English, but architect for future languages).
  - Service type: Dine‑In / Takeaway.
  - Optional table number entry or automatic ticket number.

- Ordering flow:
  - Large tile‑based menu by category (Burgers, Drinks, Desserts, etc.) with images, ratings, and badges (spicy, veg, best seller).  
  - Item detail screen with:
    - Size options, add‑ons/toppings, cooking preferences.
    - Quantity stepper, notes.
  - Smart **upsell screen** after main item selection (e.g., “Make it a combo”, “Add fries and drink”); configurable rules.

- Cart & checkout:
  - Always‑visible mini‑cart; full cart review with edit/remove items.  
  - Support promo codes and loyalty lookup (by mobile number or QR membership code).  
  - Payment step:
    - For now, simulate card/UPI/wallet payment via mock functions; design the interface to later plug in real gateways.
    - Option for “Pay at Counter” which prints or shows a token number.

- Integration:
  - Every kiosk order must go through the **same order pipeline** as POS:
    - Create Order object with channel = `kiosk`.
    - Push to KDS (by station) and inventory/recipe deduction.
    - Reflect in dashboards and reports as “Kiosk” channel.

- Device behaviour:
  - Run in “attractor loop” when idle (e.g., slideshow of offers, “Tap to start”).  
  - Auto‑reset to welcome screen after X minutes of inactivity or order completion.  
  - On-screen controls only (no browser UI).

### 1.2 Guest Web / QR Ordering

Build a **lightweight guest web app** that can be opened by scanning a QR code at the table:

- QR/URL structure:
  - Each outlet + table generates a unique, but opaque, URL (e.g., `/guest/o/{outletId}/t/{tableToken}`).
  - On open, detect outlet, table, and current service session.

- Guest experience:
  - Live menu with:
    - Availability (86’d items hidden or marked unavailable).  
    - Outlet‑specific pricing and taxes.  
    - Tags for veg/non‑veg, allergens, popular items.
  - Ordering:
    - Guests can add items to a shared table cart or per‑guest cart (for bill splitting).  
    - “Call server” button and “Request bill” action.
  - Bill & payment:
    - View current running bill (synced with POS orders already placed by staff).  
    - Choose to split bill by item or by amount for each guest.
    - For now, simulate online payment confirmation; design the API/interface so real payment gateways can be attached later.

- Integration:
  - Orders created from guest app must:
    - Appear in existing POS and KDS with channel = `qr_dinein`.  
    - Update inventory and loyalty where linked to a CRM profile.
  - Table/session management:
    - Prevent duplicate/overlapping sessions; handle table closing and re‑opening.

## 2. Omnichannel Cloud Architecture

Design and partially implement an **omnichannel layer** that unifies POS, kiosk, guest QR, aggregators, and call‑center orders.

### 2.1 Canonical Order Model

Create a **single Order schema** used by all channels:

```ts
Order {
  id: string;
  outletId: string;
  source: 'pos' | 'kiosk' | 'guest_qr' | 'aggregator' | 'phone' | 'web';
  status: 'new' | 'accepted' | 'in_kitchen' | 'ready' | 'served' | 'delivered' | 'cancelled';
  type: 'dine_in' | 'takeaway' | 'delivery';
  tableId?: string;
  customerId?: string;
  items: OrderItem[];
  totals: {
    subtotal: number;
    discounts: number;
    tax: number;
    serviceCharge: number;
    grandTotal: number;
    currency: string;
  };
  timestamps: {
    createdAt: string;
    acceptedAt?: string;
    readyAt?: string;
    closedAt?: string;
  };
  channelMeta: { /* aggregator ids, kiosk id, device id, etc. */ }
}
```

All channels must instantiate and persist this format so downstream modules (KDS, inventory, dashboards, loyalty) don’t care which channel produced the order.

### 2.2 Promotion & Pricing Rules Engine

Introduce a **rules engine** used by every channel:

- Rule types:
  - Item discount, BOGO, combo pricing, happy hour, service‑charge conditions, channel‑specific surcharges (e.g., delivery packaging fee).  
- Inputs:
  - Order context (outlet, source, time, day, customer segment, table area).  
  - Configured business rules from an admin “Promotion Center” screen.
- Outputs:
  - Adjusted line‑item prices, discount lines, service charges, and tax base.

Ensure both kiosks and guest QR flows call the same rules engine as POS and aggregators.

### 2.3 Sync and Offline‑First Behaviour

Implement a **simple but realistic sync layer**:

- Local store:
  - For POS and kiosk, use IndexedDB or localStorage to store unsynced orders and configuration snapshots (menu, prices, promotions).  
  - Queue unsent orders when network is offline; auto‑retry when back online.

- Sync service:
  - A client‑side “SyncManager” that:
    - Polls or websockets a mock “cloud” API for updates (menu changes, item availability, promotion updates).  
    - Merges updates into local caches without blocking the UI.

- Conflict handling:
  - Basic last‑write‑wins for menu and promotion updates.  
  - Duplicate order prevention using UUIDs and idempotent server calls (simulated).

Design the code with clear interfaces (`OrderRepository`, `SyncService`, `RulesEngine`) so a real backend can be swapped in.

## 3. Security, Governance, and Controls

Add a **security and governance layer** suitable for multi‑outlet and franchise operations.

### 3.1 Role‑Based Access Control (RBAC)

Implement granular RBAC:

- Roles:
  - Waiter, Cashier, Kitchen, Supervisor, Outlet Manager, HQ Admin, Franchise Owner, Auditor.  
- Permissions:
  - Actions like `create_order`, `apply_discount_above_10`, `change_price`, `void_bill`, `close_day`, `view_cost_reports`, `edit_recipe`, `approve_purchase`, etc.
- Mechanism:
  - Central `Authorization` module that exposes:
    - `can(user, action, context)` for checks inside all UIs and API calls.  
  - Store role → permissions mapping centrally (JSON or config file) and load on app start.

### 3.2 Approval Workflows & Controls

Add **approvals for high‑risk operations**:

- Scenarios:
  - Applying large discounts, voiding paid bills, changing recipe costs, making major stock adjustments, overriding promotion rules.  
- Flow:
  - User triggers high‑risk action → app checks `can()` → if not permitted, prompt for supervisor override:
    - Supervisor PIN, password, or one‑time code.  
  - Log who approved, when, and from which device/terminal.

### 3.3 Audit Logging

Implement a detailed **audit log**:

- Log events for:
  - Logins/logouts, failed logins.  
  - Order lifecycle changes (created, updated, voided, refunded).  
  - Price/tax changes, menu edits, promotion/rules changes.  
  - Inventory transactions (PO, GRN, adjustments, wastage).  
  - Role/permission changes, outlet configuration changes.

- Structure:
  ```ts
  AuditEvent {
    id: string;
    userId?: string;
    role?: string;
    action: string;
    entity: 'order' | 'menu' | 'inventory' | 'user' | 'settings' | ...;
    entityId?: string;
    before?: any;
    after?: any;
    ip?: string;
    deviceId?: string;
    timestamp: string;
  }
  ```

- UI:
  - “Audit Log” page with filters (date range, outlet, user, action, entity) and export to CSV.  
  - Per‑record history view (e.g., all changes to a specific menu item or order).

### 3.4 Session, Device, and Security Settings

Add a **Security Settings** section:

- Session controls:
  - Idle timeout, forced logout time, concurrent session limits.  
- Device binding (lightweight):
  - Mark certain browsers/devices as “trusted” POS/Kiosk terminals; restrict some actions to trusted devices only.
- Basic 2FA hooks:
  - For now, simulate OTP verification screen for manager approval flows; structure the code so a real SMS/email OTP provider can be integrated.

## Cross‑Cutting Requirements

- All new modules must:
  - Respect existing time‑zone, tax, and currency settings (use the shared configuration and formatters).  
  - Work for multi‑outlet setups and use `outletId` everywhere.  
  - Use the global `CurrencyManager` and `APP_CONFIG` you already defined.

- Architecture:
  - Organise code into modules/folders: `kiosk/`, `guest-web/`, `omnichannel/`, `security/`, `audit/`.  
  - For each module, expose clear public APIs so other parts of the app can call them.

- Demo data:
  - Seed at least:
    - 2 outlets (e.g., “Downtown QSR”, “Mall Food Court”).  
    - 1 kiosk per outlet, multiple tables for QR.  
    - Sample users with different roles for testing RBAC and approvals.

- UX:
  - Keep UI consistent with the existing “Table Salt” theme (tropical premium, aqua + sand).  
  - Make kiosk and QR flows mobile‑friendly and touch optimized.

Goal: deliver a browser‑runnable prototype where self‑service channels (kiosk + QR) work end‑to‑end through a unified omnichannel order model, with basic offline sync, shared promotions, and strong security/governance controls (RBAC, approvals, audit log), ready to be wired to a real backend later.
