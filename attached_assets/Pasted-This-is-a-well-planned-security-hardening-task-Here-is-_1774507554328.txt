## Task-by-Task Review

### Task 1 — Idempotency Key on Order Creation ✅ Correct
```
ASSESSMENT: Well designed

WHAT IS RIGHT:
  → UUID per cart session (not per app load)
  → X-Idempotency-Key header (industry standard)
  → 60-second window is correct
  → Returns existing order (not error) on duplicate
  → Clears key on success

ONE GAP TO ADD:
  → Also apply idempotency to:
    → KOT send (send to kitchen)
    → Payment processing
    → Stock adjustments
  Not just order creation

  Double-tap on "Send to Kitchen" can
  also create duplicate KOTs in kitchen
  This is equally critical
```

### Task 2 — Payment Amount Server Validation ✅ Critical and Correct
```
ASSESSMENT: Essential — good catch

WHAT IS RIGHT:
  → Re-query from database (not trust client)
  → 1 currency unit tolerance (handles rounding)
  → Log as security event (audit trail)

ONE ADDITION RECOMMENDED:
  → Also validate:
    → Discount amount not > subtotal
    → Tax amount matches configured rate
    → Tip amount is positive (not negative hack)
    → Split payment total = bill total exactly
  
  A determined bad actor could manipulate
  other fields beyond just total amount
```

### Task 3 — HTTP Security Headers ✅ Correct
```
ASSESSMENT: Standard and necessary

WHAT IS RIGHT:
  → Helmet is the right tool
  → All 5 headers are correct choices
  → CSP with nonce is proper approach

ONE CONCERN:
  → CSP with 'self' only may break
    existing features that use:
    → Inline styles (many UI libraries do)
    → External fonts (Google Fonts)
    → Razorpay/Stripe iframes
    → Chart libraries
  
  RECOMMENDATION:
  → Test CSP in report-only mode first:
    Content-Security-Policy-Report-Only
  → Fix violations before enforcing
  → Add Razorpay/Stripe to connect-src
  → Add font CDN to font-src
  → Do NOT rush CSP to production
    without thorough testing
    It is the most likely to break things
```

### Task 4 — Concurrent Session Detection ✅ Good Approach
```
ASSESSMENT: Correct and practical

WHAT IS RIGHT:
  → session_token in DB (not just JWT)
  → Mismatch = 401 SESSION_CONFLICT
  → Frontend banner (not forced logout)
  → "Stay here" re-authenticates

ONE IMPROVEMENT:
  → Current plan: checks on EVERY request
    This adds one DB lookup per request
    At 100 req/sec = 100 extra DB queries/sec

  BETTER APPROACH:
  → Check session_token only on:
    → Sensitive operations (payment, void, delete)
    → Every 5 minutes via background check
    → Not every single API request
  
  → Or: store session_token in Redis/memory
    with expiry (avoids DB hit every request)

ALSO ADD:
  → Show session list to user:
    "Currently logged in on 2 devices"
    "Log out all other sessions" button
  → This is standard on modern platforms
```

### Task 5 — Quick PIN Login ✅ Well Designed
```
ASSESSMENT: Practical for restaurant environment

WHAT IS RIGHT:
  → bcrypt hash (not plain PIN storage)
  → Nullable (PIN is optional per staff)
  → Same session mechanism as password
  → Restricted to non-admin roles
  → Owner sets PIN for staff

IMPORTANT ADDITIONS:
  → PIN lockout after 5 wrong attempts
    (same as password brute force)
  → PIN cannot be sequential (1234, 0000)
  → PIN cannot match staff ID last 4 digits
  → PIN expiry: require reset every 90 days
  → PIN change requires current PIN + manager
  → Log PIN login separately in audit trail:
    action: 'PIN_LOGIN' not 'PASSWORD_LOGIN'
  
  SECURITY NOTE:
  → 4-digit PIN has only 10,000 combinations
  → Brute force lockout is MANDATORY
    not optional for PIN login
```

### Task 6 — Global API Timeout + Retry ✅ Correct
```
ASSESSMENT: Exactly right

WHAT IS RIGHT:
  → 8 seconds timeout (reasonable for restaurant)
  → Retry only on network errors (not 4xx/5xx)
  → Exponential backoff (1s, 2s, 4s)
  → User-facing retry button

ONE ADDITION:
  → Different timeouts per operation type:
    Fast operations (menu load):    5 seconds
    Standard operations (orders):  8 seconds
    Heavy operations (reports):    30 seconds
  
  → 8 seconds is too long for a menu load
    but too short for a complex report
  
  → Configure per query key or endpoint type
```
