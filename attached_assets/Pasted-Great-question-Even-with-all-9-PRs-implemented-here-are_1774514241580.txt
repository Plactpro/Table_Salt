### CATEGORY 1 — Human Behavior Issues
#### These cannot be fixed by code

##### 1.1 Staff Training Failure
```
WHAT HAPPENS:
  Restaurant goes live Monday
  Owner trained for 2 hours on Sunday
  Chef never touched the tablet before
  Waiter uses it wrong under pressure

REAL SCENARIOS:
  → Chef taps DONE without tapping START
    → Timer never recorded
    → Performance data useless
    → Stock not deducted correctly

  → Waiter creates order without selecting table
    → Kitchen gets KOT with no table number
    → Chef shouts "which table??"
    → Service chaos

  → Cashier closes shift without reconciliation
    → Cash variance never explained
    → Owner cannot audit

  → Manager never checks wastage alerts
    → Alerts pile up unread
    → Alerts become noise
    → Real alerts ignored

CANNOT BE FIXED BY CODE
SOLUTION:
  → Mandatory onboarding checklist
  → In-app guided tour (first login)
  → Role-specific quick reference card
  → 30-day post-launch support commitment
```

##### 1.2 Deliberate Misuse
```
WHAT HAPPENS:
  → Waiter voids items and pockets cash
    difference (classic restaurant fraud)
  → Chef marks items ready without cooking
    (to clear screen during rush)
  → Manager gives own account to everyone
    (bypasses PIN login entirely)
  → Staff marks wastage for personal use
    (claims food wasted, takes it home)

YOUR SYSTEM HAS:
  ✅ Void approval required
  ✅ Audit trail on all actions
  ✅ Account sharing detection

BUT:
  → Audit trail only helps AFTER fraud occurs
  → Real-time fraud prevention needs
    pattern detection (not yet built)
  → Example: waiter voids 3+ items per shift
    consistently → automatic manager alert
```

    Real stock runs out mid-service

  → Recipe quantity wrong

YOUR SYSTEM HAS:

STILL MISSING:

    "This price is 90% lower than before"

    "500g per portion of salt seems high"
```

### CATEGORY 2 — Infrastructure Issues
#### These depend on client's setup

##### 2.1 WiFi Network Quality
```
WHAT HAPPENS:
  Your app works perfectly on good WiFi
  Restaurant WiFi is often terrible:

  COMMON SETUPS:
  → Single consumer router for entire restaurant
  → Kitchen staff WiFi + customer WiFi
    on SAME network (congestion)
  → Router placed in office, not kitchen
    (signal too weak at chef tablet)
  → No WiFi extenders in large venues
  → Router never rebooted in months

  REAL IMPACT:
  → WebSocket drops every 10 minutes
  → PR-009 WS cleanup helps server side
  → But frequent reconnects annoy staff
  → "App is broken" complaints
    but it's actually WiFi

YOUR APP HAS:
  ✅ Auto-reconnect (PR-001)
  ✅ Offline mode (PR-006)
  ✅ Connection status indicator

WHAT YOU NEED:
  → WiFi requirements document
    for restaurant clients
  → Minimum: dual-band router
    Staff on 5GHz, customers on 2.4GHz
  → Recommended: mesh network for large venues
  → This is CLIENT responsibility
    but you must communicate it clearly
```

##### 2.2 Device Quality Variance
```
WHAT HAPPENS:

  for kitchen use (₹5,000 tablets)

  PROBLEMS:
  → 1GB RAM: Chrome crashes frequently
  → Android 8: some APIs not supported
    (Wake Lock API, IndexedDB limitations)
  → Slow processor: UI lags during peak
  → Small screen: buttons too small
    for chef with gloves
  → Poor touch sensitivity: missed taps

  REAL IMPACT:
  → KDS feels "slow and broken"
  → Customer blames your software
  → Refund request

YOUR APP HAS:
  ✅ Wake Lock with graceful fallback
  ✅ Virtual scrolling for performance

WHAT YOU NEED:
  → Minimum device specification document
  → Tested device list (recommended hardware)
  → App performance on low-end device testing
```

##### 2.3 Power Cuts
```
WHAT HAPPENS:
  India/UAE: power cuts during service
  UPS covers server room but not POS tablets

  SCENARIO:
  10 open orders in progress
  Power cut for 2 minutes
  All tablets restart
  KDS wall screen restarts
  
  ON RESTART:
  → Open orders restored from IndexedDB ✅
  → KDS reloads from server ✅
  → BUT: timers reset (cooking time wrong)
  → AND: any unsaved cart data lost
  → AND: printer needs re-initialization

  YOUR APP HAS:
  ✅ Offline mode handles some scenarios
  ✅ Session recovery (PR-004 idle timer)

  REAL GAP:
  → Cart state in sessionStorage may clear
    on full device power-off restart
  → Difference: tab close vs power off
  → sessionStorage does not survive power off
  → Use localStorage or IndexedDB for cart
```

### CATEGORY 3 — Data Volume Issues
#### These appear after 60-90 days of real use

##### 3.1 Report Generation Timeout
```
TIMELINE:
  Month 1:  Reports load in 2 seconds ✅
  Month 3:  Reports load in 8 seconds ⚠️
  Month 6:  Reports timeout ❌

  YOUR APP HAS:
  ✅ Background report jobs (PR-005)
  ✅ Report cache table

  WHAT WILL STILL FAIL:
  → Reports not yet moved to background:
    → Stock movement report
    → Chef accountability report
    → Wastage analysis
    → Audit trail export
  → These will slow down at scale

  RECOMMENDATION:
  → Move ALL reports to background job pattern
    not just the 3 identified in PR-005
  → Do this before Month 2
```

##### 3.2 Audit Trail Size
```
TIMELINE:
  Every action logs to audit trail
  100 staff actions/hour
  × 12 hours/day
  × 30 days
  = 36,000 rows per month per restaurant
  × 100 restaurants
  = 3.6 million rows/month

  YOUR APP HAS:
  ✅ Audit trail exists
  ✅ Retention cleanup (30-day soft delete)

  CRITICAL GAP:
  → Audit trail should NOT be deleted
    (compliance requirement — you confirmed this)
  → But 3.6M rows/month will slow queries
  → Conflict: keep forever vs query performance

  SOLUTION NEEDED:
  → Archive old audit entries (> 1 year)
    to separate archive table
  → Keep recent 12 months in main table
  → Archive table: query separately
  → This preserves compliance
    while maintaining performance
```
