### Order Ticket History & Reopen System

This is a continuation of the existing restaurant SaaS project in Replit. The following are already built: multi-tenant architecture, POS, KDS, order management, billing, service coordination, food modifications, and time tracking.

Now build a **fully functional Order Ticket History and Reopen System** that allows authorized staff to search, view, and reopen any past order ticket — with full audit trail, controlled reopen workflows, and protection against unauthorized modifications.

#### 1. What "viewing old tickets" covers

```
WHO NEEDS TO VIEW OLD TICKETS:
  ├── Manager     → investigate complaints, audit orders
  ├── Cashier     → reprint receipt, check payment status

  ├── Chef        → reference a past preparation method

WHAT THEY NEED TO SEE:

  ├── Who did what (staff accountability trail)
  ├── Payment details (amount, method, split)
  ├── KOT and receipt history (every print job)
  ├── Time performance data (cooking, service times)
  ├── Customer details (if linked)
  └── Any modifications, cancellations, or voids
```

#### 2. When reopening is needed — real scenarios

```
SCENARIO 1: Customer returns same day
  "I was here at lunch, can I add dessert to my tab?"
  → Reopen closed bill → add item → re-bill

SCENARIO 2: Wrong item billed
  Cashier billed Butter Chicken instead of Dal Makhani
  → Reopen → correct item → reprint bill

SCENARIO 3: Payment method change
  Customer paid cash but wants to pay by card
  → Reopen payment → reverse cash → process card

SCENARIO 4: Missed item
  Waiter forgot to add one item to the order
  → Reopen order → add missed item → send to kitchen

SCENARIO 5: Customer complaint investigation
  "My food took 45 minutes" — manager needs to verify
  → View ticket timeline → confirm actual times

SCENARIO 6: Split bill after full payment
  Table of 4 paid together, now want separate receipts
  → View closed bill → generate split receipts

SCENARIO 7: Kitchen refire (remake)
  Food returned by customer — needs to be remade
  → Reopen specific item on KDS → send back to kitchen

SCENARIO 8: Partial refund
  One item was wrong, partial refund needed
  → Reopen → void specific item → adjust bill
```

#### 3. Database schema

```sql
-- Ticket history view is built from existing tables
-- Add these additional tracking tables:

-- Reopen requests and approvals
CREATE TABLE ticket_reopen_requests (
  id                  UUID PRIMARY KEY,
  tenant_id           UUID NOT NULL,
  outlet_id           UUID NOT NULL,
  order_id            UUID NOT NULL,
  order_number        VARCHAR(50),
  reopen_type         VARCHAR(30) NOT NULL,
    -- 'ADD_ITEM', 'REMOVE_ITEM', 'MODIFY_ITEM',
    -- 'PAYMENT_CHANGE', 'FULL_REOPEN', 'REFIRE_ITEM',
    -- 'RECEIPT_CORRECTION', 'PARTIAL_REFUND'
  reason              TEXT NOT NULL,
  requested_by        UUID NOT NULL,
  requested_by_name   VARCHAR(255),
  requested_by_role   VARCHAR(50),
  status              VARCHAR(20) DEFAULT 'pending',
    -- 'pending', 'approved', 'rejected', 'completed'
  approved_by         UUID,
  approved_by_name    VARCHAR(255),
  rejected_reason     TEXT,
  approved_at         TIMESTAMP,
  completed_at        TIMESTAMP,
  changes_made        JSONB,
    -- summary of what was changed after reopen
  original_snapshot   JSONB,
    -- full order state before reopen
  created_at          TIMESTAMP DEFAULT NOW()
);

-- Full audit trail of every action on every order
CREATE TABLE order_audit_trail (
  id                  UUID PRIMARY KEY,
  tenant_id           UUID NOT NULL,
  order_id            UUID NOT NULL,
  order_number        VARCHAR(50),
  action_type         VARCHAR(50) NOT NULL,
    -- 'ORDER_CREATED', 'ITEM_ADDED', 'ITEM_REMOVED',
    -- 'ITEM_MODIFIED', 'STATUS_CHANGED', 'KOT_PRINTED',
    -- 'BILL_GENERATED', 'PAYMENT_RECEIVED',
    -- 'ORDER_CLOSED', 'ORDER_REOPENED',
    -- 'ITEM_VOIDED', 'DISCOUNT_APPLIED',
    -- 'REFUND_ISSUED', 'RECEIPT_REPRINTED'
  action_description  TEXT NOT NULL,
  previous_value      JSONB,
  new_value           JSONB,
  performed_by        UUID NOT NULL,
  performed_by_name   VARCHAR(255),
  performed_by_role   VARCHAR(50),
  ip_address          VARCHAR(50),
  device_info         VARCHAR(255),
  reopen_request_id   UUID,
  created_at          TIMESTAMP DEFAULT NOW()
);

-- Voided items tracking
CREATE TABLE voided_items (
  id                  UUID PRIMARY KEY,
  tenant_id           UUID NOT NULL,
  order_id            UUID NOT NULL,
  order_item_id       UUID NOT NULL,
  menu_item_name      VARCHAR(255),
  quantity            INT,
  unit_price          DECIMAL(10,2),
  void_reason         TEXT NOT NULL,
  void_type           VARCHAR(20),
    -- 'WRONG_ORDER', 'CUSTOMER_CANCELLED',
    -- 'KITCHEN_ERROR', 'QUALITY_ISSUE', 'MANAGER_VOID'
  voided_by           UUID NOT NULL,
  voided_by_name      VARCHAR(255),
  approved_by         UUID,
  is_refunded         BOOLEAN DEFAULT false,
  refund_amount       DECIMAL(10,2),
  created_at          TIMESTAMP DEFAULT NOW()
);
```

***

Ensure all reopen actions are wrapped in DB transactions — partial changes must never be saved. Every single change during a reopen session must write to `order_audit_trail` immediately. The `original_snapshot` must be saved to `ticket_reopen_requests` before any changes are made so the order can always be restored. Index `order_id`, `action_type`, `performed_by`, `created_at`, and `tenant_id` on `order_audit_trail` for fast history queries. Never hard-delete any order data — use soft deletes and void flags only.
