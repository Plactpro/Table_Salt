## BUG #001 — CRITICAL
**Title:** Ticket History — "View" button throws "Something went wrong" error

**Module:** Ticket History (`/tickets`)
**Severity:** 🔴 Critical
**Priority:** P1

**Steps to Reproduce:**
1. Log in as Owner or Manager
2. Navigate to `/tickets` (via POS → History tab or sidebar)
3. Click the **View** button (→ arrow) on any ticket in the list

**Expected Result:** Ticket detail modal or page opens with full order information

**Actual Result:** A full-page "Something went wrong" error is displayed. The app crashes and the user must manually navigate back.

**Impact:** Staff cannot review historical orders from the Ticket History screen. Affects Owner, Manager, and Waiter roles.

**Suggested Fix:** Check the API call triggered by the View button — likely a missing or malformed parameter (e.g., ticket ID not being passed correctly to the backend endpoint). Add error boundary and fallback UI.

## BUG #002 — CRITICAL
**Title:** Online Orders list — Advancing order status fails with "Invalid CSRF token"

**Module:** Online Orders (`/orders`)
**Severity:** 🔴 Critical
**Priority:** P1

**Steps to Reproduce:**
1. Log in as Owner or Manager
2. Navigate to `/orders`
3. Click the **→ (advance status)** arrow button on any In Progress order

**Expected Result:** Order status advances to the next stage (e.g., In Progress → Ready)

**Actual Result:** Error toast: `"Failed to update — Invalid CSRF token"`

## BUG #003 — HIGH
**Title:** Delivery — "Mark Delivered" throws Error 500: `e.toISOString is not a function`

**Module:** Delivery & Online (`/delivery`)
**Severity:** 🟠 High
**Priority:** P1

**Steps to Reproduce:**
1. Log in as Owner or Manager
2. Navigate to `/delivery`
3. Find any order in the **"Out for Delivery"** kanban column
4. Click the **"Mark Delivered"** button

**Expected Result:** Order status updates to "Delivered", moves to the "Delivered Today" column

**Actual Result:** Error 500 displayed: `"e.toISOString is not a function"`

**Impact:** Delivery agents and managers cannot mark orders as delivered from the delivery board. Delivery status tracking is broken. Confirmed broken for delivery agent role as well.

**Suggested Fix:** A variable expected to be a `Date` object is a `string` or `null` when `.toISOString()` is called on it. Likely in the delivery status update handler — ensure `deliveredAt` or similar timestamp field is properly instantiated as `new Date()` before calling `.toISOString()`. Example fix:
```javascript
// Before (broken):
const delivered = someField.toISOString();

// After (fixed):
const delivered = new Date(someField).toISOString();
// or
const delivered = new Date().toISOString(); // if it's "now"
```

## BUG #004 — HIGH
**Title:** Procurement — Create Purchase Order fails with Error 400 when "Expected Delivery" date is empty; no frontend validation shown

**Module:** Procurement (`/procurement`) → Purchase Orders → New PO
**Severity:** 🟠 High
**Priority:** P2

**Steps to Reproduce:**
1. Log in as Owner or Manager
2. Navigate to `/procurement` → Purchase Orders tab
3. Click **+ New PO**
4. Select a supplier (e.g., Farm Direct Produce)
5. Add a line item with quantity and unit price
6. Leave **Expected Delivery** date empty
7. Click **Create PO**

**Expected Result:** Frontend should show a validation error: "Expected Delivery date is required"

**Actual Result:**
- API call is made to the backend
- Error 400 returned: `{"message": "[{\"code\": \"invalid_type\", \"expected\": \"date\", \"received\": \"string\", \"path\": [\"expectedDelivery\"], \"message\": \"Expected date, received string\"}]"}`
- Raw JSON error toast shown to user — not user-friendly
- No form-level validation preventing submission

**Impact:** Users see a raw technical JSON error instead of a helpful message. The form allows submission without a required field.

**Suggested Fix:**
1. Add frontend required field validation on the `expectedDelivery` date input before form submission
2. If the API returns a 400 for this field, map it to a human-readable message: "Please select an expected delivery date"
3. Mark the date field with a red asterisk (*) as required in the UI

## BUG #005 — HIGH
**Title:** Inventory — Add Item throws Error 500: "Invalid input syntax for type integer"

**Module:** Inventory (`/inventory`) → Stock & Items → Add Item
**Severity:** 🟠 High
**Priority:** P2

**Steps to Reproduce:**
1. Log in as Owner or Manager
2. Navigate to `/inventory`
3. Click **+ Add Item**
4. Fill in: Name = "Black Truffle Shavings", SKU = "BTS-001", Category/Tag = "Specialty", Current Stock = 2, Cost Price = 450, Reorder Level = 1, Supplier = "Gourmet Direct" (free-text)
5. Click **Save / Create**

**Expected Result:** New inventory item created successfully

**Actual Result:** Error 500: `"Invalid input syntax for type integer"`

**Impact:** New inventory items cannot be added. Stock management is partially broken.

**Suspected Cause:** The `supplier` field in the Add Item form is a free-text input, but the database schema expects a supplier `ID` (integer foreign key). The string "Gourmet Direct" is being sent directly to an integer column. Similarly, the SKU field "BTS-001" may be conflicting with an integer column.

**Suggested Fix:**
1. Replace the free-text supplier field in the Add Item modal with a dropdown linked to the existing suppliers table (same as in the Create PO form)
2. Validate SKU as alphanumeric string (not integer) in the schema
3. Add proper error handling to return a user-friendly message instead of raw DB error

## TESTING NOTES FOR REPLIT AI

1. **CSRF Token (#002):** The CSRF token issue on `/orders` status update suggests the token is not being refreshed or passed correctly on that specific API call. Compare the working KDS status update implementation with the Orders page one — the fix is likely a one-line addition of the token to the request header.

2. **Date/ISO String Bug (#003):** The `toISOString` crash is a classic JS type error. Search the codebase for `.toISOString()` calls in delivery-related functions and add a `new Date()` wrapper.

3. **Inventory Supplier FK (#005):** The Add Item modal should reuse the same supplier dropdown component used in the Create PO form. The backend schema clearly expects a foreign key integer for supplier, not a free-text string.

4. **All raw API errors (#004, #005):** Consider adding a global error handler that intercepts 400/500 responses and displays a user-friendly toast instead of raw JSON. This will improve UX across the entire application.
