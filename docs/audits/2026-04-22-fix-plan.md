## C1 — Timezone off-by-one (Critical)

### Evidence

- Tables → Reservations → "Reservation date automatically reserved to the next day while entering"
- Tables → Reservations → "reservation made on 16 April 2026 is updated to 17 April 2026" (confirms the same bug twice, on different days)
- Events → "When the start date and end date of an event fall on the same day, the event is incorrectly displayed on the next day in the calendar"
- Staff & Workforce → Schedule → Shift → "shift created on 21 April 2026 is updated to 22 April 2026"
- Related but not identical: POS → "when we print the bill the time is incorrect even though changed in settings"

### What it probably is (HIGH confidence)

Four bugs, same class. All show "today's date saves/renders as tomorrow." The shared pattern is the client sending a `YYYY-MM-DD` date to the server which interprets it as UTC midnight, which becomes the next day in any timezone ahead of UTC (AED is UTC+4, IST is UTC+5:30). Or the reverse — a `Date` object created client-side without timezone normalization. The F-225 work that's already in flight relates to this same area (`fix/F-225-tenant-tz-helper` branch still exists locally).

The POS "bill time incorrect" bug may belong here or may be a separate issue (settings-layer vs render-layer); recon should clarify.

### Shape of the fix

Two possible shapes:
- **If the bug is in one shared date helper:** a 10–30 line fix that normalizes dates to tenant timezone before storing / after loading. Touches maybe 2–3 files.
- **If each writer does its own date handling:** a larger fix touching reservations.ts, events.ts, shift scheduler — each getting the same treatment. Probably 100–200 lines.

### Recon required before code

1. `git log` `fix/F-225-tenant-tz-helper` to understand what that branch was doing and why it stalled. There may already be a helper that's half-built.
2. Grep for `new Date(` and `setHours(` and `.toISOString()` in `server/routers/reservations.ts`, `events.ts`, and shift scheduling code.
3. Check whether the four affected writers share any common date parsing code. If yes, that's the fix point.
4. Identify the tenant's timezone setting (the replit.md mentions IANA TZ support — confirm it's populated).

### Risk

- Touching timezone code is where projects create "we fixed it in one place and broke three others" regressions. The F-225 stall may have happened because the full scope became clearer only mid-work.
- A bad fix here silently shifts historical data display for every tenant.
- Mitigation: write one automated test case per writer with a fixed "now" timestamp before editing any logic.

### Why Critical

Reservations saving for the wrong day is a direct customer-facing problem. Staff book a table for tonight, customer arrives, there's no reservation showing. This alone could be the difference between "I'd recommend Table Salt" and "it lost my booking."

## C2 — Tenant guard 500 on bill path (Critical)

### Evidence

- Observed by Arun directly on 2026-04-22: red 500 toast on Bill Preview path, message `[TENANT_GUARD] getOrderItemsByOrder called without a valid tenantId (got undefined) — this is a bug, not a user error`

### What it probably is (HIGH confidence it's real, LOW confidence on root cause)

The toast text is specific enough — it's a real error surfaced from server code, not a generic "something went wrong." The message is explicit: a function called `getOrderItemsByOrder` is being called without `tenantId`, and a tenant guard is correctly blocking the call and logging the error. The fact that the guard message ends with "this is a bug, not a user error" tells us the codebase already has tenant-scoping plumbing and one code path is missing it.

Without code access I can't narrow further, but the shape is: some bill preview / ready-to-pay route is fetching order items via a storage method that expects `tenantId`, and either the route handler is not extracting `tenantId` from the session or it's a public route that shouldn't be calling this at all.

Possible neighbors (tester bugs that may have the same root cause):
- Online Orders → Ready to Pay → "CRM is not working for newly created customer" (could be same tenantId-missing shape)
- Online Orders → Ready to Pay → "Proceed to Payment does not navigate" (could be a side-effect of the 500)
- Online Orders → Ready to Pay → Print Preview → "displays that payments are made in cash for every order" (data bug, not the same — but worth checking)

### Shape of the fix

Once recon identifies which route is calling `getOrderItemsByOrder` without `tenantId`, the fix is likely 5–20 lines: extract `tenantId` from `req.user.tenantId` or session, pass it to the storage call. If it's a public route (e.g., `/api/public/receipt/:id`), the fix is to derive `tenantId` from the bill record itself (we did exactly this kind of fix in F4 on commit `a024e11` — public receipt used `user.tenantId` when it should have used `bill.tenantId`).

### Recon required before code

1. Grep `server/` for `getOrderItemsByOrder` — find all callers.
2. For each caller, trace back: where does `tenantId` come from? Does any caller pass `undefined`?
3. Arun to reproduce the error and capture the Network tab response so we know the specific endpoint URL.

### Risk

- If recon shows the `tenantId` is missing from a session middleware (not a per-route oversight), the fix could be a middleware change affecting many endpoints. Broader blast radius.
- If we guess at the fix without repro, we could silently mask errors rather than fix them.

### Why Critical

This is a 500 on the payment path, which is the most brittle and highest-stakes code path in a restaurant system. Any 500 between "customer wants to pay" and "payment succeeds" is customer-facing damage.

## C3 — Payment flow breakage (Critical)

### Evidence

- POS → "Takeaway — before sending it to the kitchen it asks the customer to pay. Once paid, the payment page opens again"
- Phone Orders → Save Draft → "it should save as draft but it sends to kitchen"
- Online Orders → Ready to Pay → "Proceed to Payment does not move to payment page"

### What it probably is (HIGH confidence these are three separate bugs, not one)

1. **Takeaway double-pay** — probably a missing state transition. The takeaway flow asks for payment, receives payment, then re-opens payment modal. Likely: the payment modal's `onSuccess` handler is not clearing the "needs payment" state, or a stale `paymentStatus` is re-triggering the modal. Similar failure mode to issues we've seen in `pos.tsx` this week.
2. **Save Draft fires kitchen** — the Save Draft button is wired to the wrong mutation, or both the draft action and the "send to kitchen" action run in parallel. One-liner likely.
3. **Proceed to Payment does not navigate** — probably blocked by the same M6 tenant-guard 500. Recon should verify: if the 500 happens before navigation, Proceed-to-Payment would visibly "do nothing." Start with M6 and this may resolve as a side effect.

### Shape of the fix

- Takeaway double-pay: ~30-line fix in POS payment flow. Test: place takeaway order, pay, verify the payment modal closes and does not reappear.
- Save Draft: ~5-line fix. Trace the button's onClick. Test: save a draft, verify no KOT fires in kitchen print queue.
- Proceed to Payment: likely resolves when M6 resolves. Verify after M6 ships.

### Recon required before code

- Takeaway: read `client/src/pages/modules/pos.tsx` around the takeaway payment handlers. We already know this file intimately from M1/M3 work — probably 20 min.
- Save Draft: find the `phone-orders.tsx` or equivalent component, trace the Save Draft button handler.
- Proceed to Payment: ship M6 first, retest.

### Risk

- Payment-flow regressions are the class of bug that makes Arun wake up to "we double-charged a customer" calls. Move carefully. Each of these should get its own narrow test before shipping.

### Why Critical

If Takeaway is actually double-prompting for payment in production, and staff are pressing Pay again thinking the first attempt failed, there's a non-zero chance of double charging. That's catastrophic.

## C4 — Unit conversion in kitchen (Critical)

### Evidence

- Kitchen & Stock → Recipe → "quantities entered in units such as tsp or tbsp are incorrectly displayed as kilograms on the kitchen board"
- Kitchen & Stock → Wastage Control → "wastage quantity entered in grams is incorrectly being treated as kilograms. (For example — the cost of lamb rack is AED 22/kg, so 100 g of lamb rack is AED 2.20, but the wastage cost is displaying as AED 22,000.00)"

### What it probably is (HIGH confidence)

One root cause, two symptoms. The codebase has a shared unit-conversion utility that treats any numeric value as kilograms, or a recipe/wastage rendering component is dropping the unit and assuming kg. The wastage example is unambiguous: displayed cost is exactly 1000× expected, which is the g→kg conversion factor.

### Shape of the fix

Likely one of:
- A shared `convertToBaseUnit(value, fromUnit, toUnit)` helper exists but is being called wrong, or
- No helper exists, and two separate code paths both do math without unit awareness.

Either way, fix size is 30–80 lines: introduce or correct the conversion helper, add a unit test, update the two call sites (wastage, recipe kitchen-board render).

### Recon required before code

1. Grep for `"tsp"`, `"tbsp"`, `"kg"`, `"g"` across `server/` and `client/` to find unit-handling code.
2. Check whether there's a unit enum or constants file already.
3. Find the wastage cost calculation — it's either on the server (cost aggregation) or the client (display only). Likely server.

### Risk

- Unit conversion bugs in a financial context (wastage cost displayed as AED 22,000) could cause mis-reporting to managers who make real purchase decisions. Fix carefully.
- Regression risk: any existing recipe that had a unit workaround applied (staff manually entering "0.100" kg for 100g) would suddenly display double-counted.

### Why Critical

AED 22,000 wastage cost on AED 2.20 of actual waste is the kind of number that would get reported to the owner and treated as theft. This is a category-of-damage bug even without customer exposure.

## C5 — CSRF on 3 write endpoints (Critical)

### Evidence

- Parking → Valet Staff → Log Key Action — CSRF token error
- Parking → Shifts → Add Staff to Shift — CSRF token error
- Staff & Workforce → Leave → Request Leave — invalid CSRF token error

### What it probably is (MEDIUM confidence)

Three write endpoints, all fail with "invalid CSRF token." This strongly suggests:
- Either a middleware order issue (CSRF check runs before session middleware on these routes), or
- These three routes were added after CSRF protection was introduced and they're missing the CSRF token header in their frontend fetch calls, or
- These routes use a different fetch wrapper (older `fetch` vs newer `apiRequest`) that doesn't inject the CSRF header.

Confidence MEDIUM rather than HIGH because the mechanism differs depending on which of the three possibilities is at play. But the **fix size is likely small in all cases** — a configuration or a header injection.

### Shape of the fix

- If middleware order: 1–2 lines in `server/index.ts`.
- If missing header in fetch: 3 × 1-line fixes in the three frontend components calling these endpoints.
- If wrong fetch wrapper: find/replace across 3 components.

In all cases, <30 line change.

### Recon required before code

1. Test one of the three write paths as Arun in browser, inspect Network tab. Is the request missing `x-csrf-token` header, or is the server rejecting a token that is present?
2. If header missing: find the button's onClick, check which fetch wrapper it uses.
3. If header present but rejected: check server CSRF middleware for route scoping.

### Risk

- Low. CSRF fixes are narrow. Risk is only shipping a fix that appears to work but actually weakens CSRF protection (e.g., disabling CSRF on these routes rather than fixing it).

### Why Critical

Staff cannot submit leave requests. Managers cannot add valet staff. These are high-frequency tasks blocked entirely. Business operations are impaired.

## C6 — Order-source parity (High)

### Evidence (five bugs, same cluster)

- Phone Orders — "when placed orders through phone order the item does not display in Online Orders"
- Phone Orders → Delivery — "when an order is made through delivery it does not display in Delivery and Online Orders"
- Delivery & Online → Online Orders → "Ordered items are not displayed for orders from Swiggy, Zomato, Uber Eats, Kiosk, QR orders, and Phone Orders. Only POS orders show the items. Ordered items is also not visible in kitchen"
- Delivery & Online → Online Orders → "Only kiosk orders with Pay at Counter are displayed here; orders with other payment modes are not shown"
- Delivery & Online → Online Orders → "When live orders are sent to the kitchen, the Order ID differs from the one shown in the Live Orders"
- Delivery & Online → Online Orders — "The Cancel option is not functional"

### What it probably is (MEDIUM confidence)

Biggest single cluster by impact. Pattern: orders from any source other than POS get dropped / mis-rendered / mis-identified somewhere in the pipeline. The code treats POS orders as first-class and every other channel (phone, kiosk, QR, aggregators) as second-class.

Possibilities:
- A shared `displayOrderItems()` component reads items from a POS-specific field (`pos_order_items`) rather than the unified `order_items` table.
- Kitchen routing logic generates a new internal order ID but doesn't surface the external/channel order ID in the live orders list.
- Online Orders filtering logic on the backend has a hardcoded `WHERE payment_method = 'at_counter'` clause for kiosk.

### Shape of the fix

Largest cluster. Easily 2–4 branches of work:
- Sub-cluster 1: phone orders not appearing in Online Orders / Delivery feed — feed filter fix, probably 1 line.
- Sub-cluster 2: non-POS orders hiding their items — render component fix, 30–80 lines.
- Sub-cluster 3: kiosk payment mode filter — single-line fix.
- Sub-cluster 4: order ID inconsistency — schema-level decision needed; could require a migration.
- Sub-cluster 5: Cancel button — probably a wiring fix in online-orders.tsx.

### Recon required before code

This cluster needs the most recon. At least 2–3 hours to understand:
1. How are orders from each channel stored? (shared `orders` table or separate?)
2. How does the Online Orders page filter and render?
3. Which component renders the item list, and what does it query?
4. What's the difference between the "Order ID in live orders" and "Order ID in kitchen"?

Only then design the fix(es).

### Risk

- Highest risk in this plan. Changes to the orders pipeline can break the four fixes we shipped today (B1a/B1b/M3/M1 all touched or depended on `orders` table semantics).
- Strong case for doing this cluster last, after the simpler clusters are closed and we have time to move carefully.

### Why High not Critical

These bugs are customer-facing but not data-destructive. An order from Swiggy may not show items, but the order itself is presumably placed correctly and can still be fulfilled. The `Cancel` button not working is annoying; the kitchen having a different Order ID is confusing but not catastrophic. Arun and testers didn't flag any of these as "we lost money" — they're parity and UX bugs, not billing bugs.

## C7 — Table operations (High)

### Evidence

- When merging two tables, the total capacity is not updated correctly (merging two tables of 4 should result in capacity 8, but system shows 4). Additionally, after merging, only one table is marked as occupied, the other remains available.
- Tables → Clear — "clearing the table only works some of the time"
- POS → "Reserved tables are not able to select when the reserved customer arrives for dine-in"

### What it probably is (MEDIUM confidence)

Three bugs in table state management. Each probably independent:

1. **Merge broken** — merge endpoint updates one table's state but misses the second (and doesn't sum capacity). Likely a missed `UPDATE` statement or a transaction that partially succeeds.
2. **Clear intermittent** — the "sometimes" wording is telling. Probably a race condition between the clear endpoint's UPDATE and a WebSocket-driven refetch that overwrites the cleared state with stale data. We just added a `table:updated` subscriber in M1 today — that may intersect with this.
3. **Reserved not selectable** — the POS table picker disables reserved tables, but there's no "seat the reservation" flow that would transition a reserved table into occupied. This was flagged in today's POS audit (section 2.5).

### Shape of the fix

- Merge: rework merge endpoint. 50–100 lines. Medium scope because it involves two-table atomic update.
- Clear: investigate WebSocket timing. 20–50 lines. Or simpler: add a short optimistic update delay before the WS handler fires invalidation.
- Reserved-to-occupied: new UI flow. Either (a) allow selecting reserved tables with a confirmation prompt, or (b) add an explicit "Seat Reservation" button that transitions state. 80–150 lines.

### Recon required before code

1. Read `server/routers/tables.ts` merge endpoint. Check the UPDATE statements.
2. Observe clear-table timing in production logs while a tester exercises it. Is there always a WS event that races the state?
3. For the reserved/occupied transition, talk to Arun about product preference (prompt vs new button).

### Risk

- Medium. Merge bug could cause data inconsistency (two "linked" tables diverge). Test carefully before shipping.

### Why High

Merge not working correctly means the system double-books the "free" half of a merge. That's a real customer-impact bug. But it's not data-destructive, and a workaround exists (don't use merge).

## C8 — QR ordering (High, LOW-MED confidence on root cause)

### Evidence

- Tables → QR code → "When placing an order through the QR code, clicking on All does not display all menu items. However, after selecting each category and then clicking All, all menu items are displayed"
- Tables → QR code → "If items are ordered and the order is placed successfully, clicking on View Bill displays different items that were not ordered. The ordered items and the items shown in the bill do not match"
- Tables → QR codes — "failed to generate QR code error"

### What it probably is (LOW-MED confidence)

Three separate QR-related bugs. The "View Bill shows wrong items" is the alarming one — could be a tenant-isolation bug (showing another table's bill) or a stale-cache bug. Without seeing the code path, I can't say which.

QR code generation failing is probably a generation-library config issue.

"All" tab showing nothing on first render is probably a data-loading race — the first render happens before menu categories are loaded.

### Shape of the fix

- "All" not showing items: 5–15 line fix in QR page component — ensure menu is loaded before "All" selection.
- View Bill mismatch: depends entirely on cause. Could be 10 lines or 100.
- QR generation error: library-level investigation, 30-60 min.

### Recon required before code

1. Reproduce each of the three bugs in a QR ordering session on a test tenant.
2. Capture Network traffic for the View Bill mismatch especially — compare order items vs what's rendered.
3. Look for tenantId scoping on the QR endpoints — this is a public-facing surface.

### Risk

- **If "View Bill mismatch" is a tenant-isolation bug, it's a data-leak security issue, not just a UX bug.** That escalates the severity.
- QR endpoints are the public-facing surface mentioned in bug-inventory as "A-14 QR public surface audit pending." This cluster overlaps with that. If A-14 hasn't been done, this cluster is a good reason to do it.

### Why High (potentially Critical pending recon)

If View Bill shows another tenant's items, that's Critical. If it shows the same tenant's different table's items, that's High. If it shows the right tenant and table but wrong items due to a local caching bug, that's Medium. Recon required.

## C9 — Currency render leaks (High)

### Evidence

- Online Orders → Ready to Pay → "the amount is shown with the rupee symbol in both the bill preview and the payment print options"
- Reports & Analysis → Shift Reconciliation — "all the amount values are displayed in dollars (only in Shift Reconciliation)"

### What it probably is (HIGH confidence)

F15 shipped on 2026-04-09 fixed the settings/data layer (outlets stored as INR/India, should have been AED/UAE). F15 did not touch the render layer. These two reports are render-layer hardcoded currency symbols, probably `₹` and `$` literals in JSX, not flowing through a `formatCurrency()` helper tied to tenant currency.

### Shape of the fix

Find & replace. Identify all hardcoded currency symbols (`₹`, `$`, `AED`, `USD`, `INR`) in the client codebase, route each through a `formatCurrency(amount, currencyCode)` helper, pass tenant currency from context.

Probably 100–200 lines across 5–10 files. Each file-level change is small; the cumulative change is larger.

### Recon required before code

1. Grep for `₹` and `$` in `client/src/**/*.tsx` to find all hardcoded currency symbols.
2. Check whether a `formatCurrency()` helper already exists.
3. Check whether tenant currency is already in a React context.

### Risk

- Low. Currency rendering is visual-only; a bad fix just shows the wrong symbol, which is easy to spot in QA.

### Why High

A UAE restaurant showing dollars on their reconciliation report is both embarrassing and operationally broken (the manager can't reconcile to AED cash counts). Same for bill preview showing ₹.

Probably one branch for "overflow everywhere", and a second branch for "categories should filter."

### Recon required before code

- 15 min to find the standard dropdown component in use and understand why it's not applied consistently.

### Risk

- Low. CSS and UI-behavior fixes.

### Why Medium

Annoying usability issue, not data or billing. Testers flagged it enough times that it's a quality signal, not a corner case.

### Risk

### Why High

## What I want Arun to notice in this plan

1. **Cluster-based ordering is cheaper than severity-based ordering.** Doing C1 in one branch fixes four tester-reported bugs at once. Doing four "Critical" bugs by tier means four separate recon passes and four branches.

2. **Some "Critical" bugs are actually tiny fixes** — C5 (CSRF) is likely a ~10-line middleware change. Shipping it in 1 hour tomorrow would reduce tester frustration disproportionately to the work involved.

3. **The big scary cluster (C6 order-source parity) should wait.** Not because it doesn't matter but because it touches the most code, carries the most risk, and every day we spend on C5 / C4 / C2 instead gets us a cleaner baseline to make C6 changes against.

4. **M1b (server-side table-claim race)** is not in these clusters because it's already separately tracked. It belongs between Week 1 and Week 2 as its own branch.

5. **The F-225 stalled branch** may contain half-done work on C1. Recon for C1 should start there.

6. **Shared tester concerns that aren't in this plan** — testers repeatedly flag the "wherever dropdown is present, scroll should work" pattern, and cosmetic alignment issues across many screens. Neither is shippable as a single fix — both are quality trajectories, not point bugs. Worth acknowledging to testers that the pattern is captured even though no single cluster solves it.
