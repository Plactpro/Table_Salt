## Decision tree — which hypothesis is correct?

Each branch lists two production observations: one that **confirms** the hypothesis and one that **rules it out**. None of these have been verified; the decision tree is purely a triage map for Path B.

### H1 — Initial-render race

**Premise:** `tenant` is `undefined` while `GET /api/tenant` is in flight. The Settings page Select renders with the `useState("USD")` initial value for that frame, then re-renders to `"KRW"` once `useEffect` fires from `setCurrency(tenant.currency || "USD")`.

**Confirms H1:**
- Tester observes the dropdown briefly showing **"$ US Dollar (USD)"** for a fraction of a second on first paint (Cold cache only — first time the page loads after login or full refresh), then it flickers to "₩ South Korean Won (KRW)".
- React DevTools shows `currency` state transitioning from `"USD"` → `"KRW"` after the `useQuery` resolves, with `tenant` initially `undefined` then populated.
- Network panel shows `GET /api/tenant` returning `200` with `currency: "KRW"` in the response body.

**Rules out H1:**
- No flicker observed at all, dropdown is stable from first paint.
- Cold-cache page loads consistently show the correct value (KRW) with no USD frame.
- `currency` state is `"KRW"` immediately on mount per React DevTools (no `"USD"` → `"KRW"` transition).

**Severity if confirmed:** medium — only visible on cold loads and only for a fraction of a second.

### H2 — Stale cached response

**Premise:** Global `staleTime: Infinity` (`queryClient.ts:202`) means the Settings page's `["/api/tenant"]` query, once populated, is never refetched on remount. If the cache holds an OLD value from a previous fetch (e.g., before the user saved KRW in another tab or before a server-side change), navigating back to Settings shows the stale value. The mutation's `invalidateQueries` only fires after a save in the same tab.

**Confirms H2:**
- Tester observes correct value (KRW) immediately after save in tab A, then opens Settings in tab B (same browser session) and sees the old value (USD) without any flicker.
- Hard refresh (Ctrl+Shift+R) in tab B fixes it; soft remount via in-app navigation does not.
- Network panel in tab B shows NO `GET /api/tenant` request when navigating to Settings (cache hit, no refetch).

**Rules out H2:**
- Network panel always shows a fresh `GET /api/tenant` on every Settings mount.
- The bug reproduces in a single-tab session with no cross-tab interaction.
- Hard refresh and soft refresh produce the same wrong dropdown state.

**Severity if confirmed:** medium-high — hits any user with a multi-tab workflow.

### H3 — Tester report imprecision

**Premise:** The "blank" the tester reports may actually be the default "$ US Dollar (USD)" rendering, which the tester read as blank because the trigger has no obvious selected-state styling. The Preview card showing `$ 1234.56` is also consistent with `currency === "USD"`. Under this hypothesis there is no bug — the dropdown shows USD and the rest of the system uses KRW because the auth-context query (`auth.tsx:101-144`) is normalizing the response while the Settings page query is reading `tenant.currency` directly.

**Confirms H3:**
- Tester clicks the Currency dropdown trigger and sees the highlighted item is "USD" (not nothing).
- The "blank" trigger area actually contains text "$ US Dollar (USD)" upon close inspection.
- React DevTools shows `currency` state is `"USD"` (the default), not `""` or `undefined`.

**Rules out H3:**
- The trigger renders genuinely empty — no text, no symbol, no padding visible. Inspecting the DOM shows an empty `<span>` inside the trigger.
- React DevTools shows `currency` state is `""` or `undefined`, not `"USD"`.

**Severity if confirmed:** trivial — UX clarity issue, not a bug. Documentation/labelling fix only.

### H4 — Cache hydration order

**Premise:** Two queries hit `/api/tenant`: the auth context's normalizing query at `auth.tsx:101-144` (key `["/api/tenant", user?.tenantId]`, custom queryFn that defaults missing fields), and the Settings page's raw query at `settings.tsx:222` (key `["/api/tenant"]`, default queryFn). They share the same key prefix. If the auth context's normalized data leaks into the Settings page's cache slot via TanStack Query's prefix-match invalidation behavior, the Settings page could read a normalized object where `currency` got defaulted to `"USD"` by the auth queryFn — even though the raw server response was `"KRW"`.

**Confirms H4:**
- React DevTools shows the Settings page's `tenant` value has fields shaped like the auth-context's `TenantInfo` (e.g., `subscriptionStatus`, `trialEndsAt`, `defaultLanguage` populated) rather than the raw GET response.
- React Query DevTools shows the cache for key `["/api/tenant"]` holding a normalized object identical to the auth-context's output.
- The dropdown blanks specifically when the auth context's query fires before the Settings page's query (timing-dependent).

**Rules out H4:**
- Settings page's `tenant` matches the raw GET response exactly (no `subscriptionStatus`, no `defaultLanguage`).
- React Query DevTools shows two separate cache slots with different shapes for `["/api/tenant"]` vs `["/api/tenant", "<tenantId>"]`.

**Severity if confirmed:** high — architectural duplication causes silent data corruption; will recur on other settings.

## Single-cause vs combined-cause

These hypotheses are not mutually exclusive. H1 and H2 can co-exist (race on cold load, stale cache on warm remount). H4 if true would mask H1's symptoms because the cache returns a non-undefined value immediately. Path B should not stop at the first confirmed hypothesis — the recon checklist below is structured to falsify or confirm each independently.

## Fix candidate drafts

Plain-language descriptions only. NO code yet — actual implementation happens after Path B confirms which hypothesis is correct.

### Fix candidate for H1 (initial-render race)

**Files touched:** `client/src/pages/modules/settings.tsx` only.

**Shape:** Gate the Settings form render on the `isLoading` flag from the existing `useQuery` (line 222 already destructures `isLoading`). Do not render any of the dropdowns (or the entire form section) until tenant data has loaded. Show a skeleton or spinner during the loading frame.

**Trade-off:** brief loading state visible to user (a few hundred ms on cold load). Prevents the USD-flicker.

**Alternative shape:** initialize the `useState` calls with `null` or `undefined` instead of `"USD"`, and use a derived render that reads from `tenant?.currency || "USD"` directly. Slightly tighter but couples render logic to the data source.

**Risk:** low. Single-file change, well-bounded.

### Fix candidate for H2 (stale cached response)

**Files touched:** `client/src/pages/modules/settings.tsx` only (most likely), or `client/src/lib/queryClient.ts` if we change global defaults (NOT recommended).

**Shape:** Override `staleTime` on the Settings page's `["/api/tenant"]` query to a finite value (e.g., `staleTime: 0` to always refetch on mount, or `5 * 60 * 1000` for 5 minutes). Optionally also set `refetchOnMount: "always"` for explicit refetch behavior.

**Trade-off:** more `GET /api/tenant` requests per session. The endpoint is cheap (single row by tenantId) so the cost is minimal.

**Alternative shape:** Keep `staleTime: Infinity` but add `refetchOnMount: true` so each Settings page mount triggers a fresh fetch even if the cache is "fresh" by staleTime. This narrows the change to remount behavior only.

**Risk:** low. Confined to one query's options.

### Fix candidate for H3 (tester imprecision)

**Files touched:** none for the bug itself.

**Shape:** No code change. Document the observed behavior in `audit/00-backlog.md` and close F-256b as "not a bug, UX clarity follow-up." Optionally improve trigger styling in `client/src/components/ui/select.tsx` so the selected value is visually distinct from a placeholder/empty state, but that is a sweep across all Selects in the app and out of scope for F-256b.

**Trade-off:** if H3 is correct, the user-perceived "bug" remains visible until a separate UX pass.

**Risk:** zero code risk. Communication risk if testers feel dismissed.

### Fix candidate for H4 (cache hydration order)

**Files touched:** `client/src/pages/modules/settings.tsx` and possibly `client/src/lib/auth.tsx`.

**Shape A — consolidate to the auth-context query:** remove the Settings page's `useQuery({ queryKey: ["/api/tenant"] })` entirely. Read tenant data via `useAuth().tenant` instead. The auth-context query has normalization (which is what we want) and lives at a cleaner architectural layer. Change ~1 line of useQuery → ~1 line of useAuth destructuring; the rest of the file works unchanged because field names match.

**Shape B — consolidate to the raw query:** remove the auth-context's normalizing query. Have AuthProvider read from a plain `useQuery({ queryKey: ["/api/tenant"] })` and let consumers handle defaults locally. More invasive — affects every component that reads `useAuth().tenant`. NOT recommended unless H4 is confirmed AND Shape A reveals other coupling issues.

**Trade-off (Shape A):** slight behavior change — Settings page now reads normalized data with defaulted fields. For currency this is identical (both default to "USD"). For other fields (e.g., `subscriptionStatus`, `defaultLanguage`) the Settings page may now expose values it did not before; verify this does not break any existing display logic.

**Risk:** Shape A is medium-low. Shape B is high — touches many components.

## Combined-cause fix (defensive)

If Path B reveals two or more hypotheses are simultaneously true, a defensive single-PR fix could combine H1's loading gate + H2's `refetchOnMount: true` + H4 Shape A's consolidation into a single change. This is the maximum-coverage fix and is the only one that closes the bug regardless of which hypothesis is the "real" one. Recommended only if recon time is limited and we cannot pin down the precise cause.

## Recon checklist for Path B

This section is the deliverable that goes to testers when Path B kicks off. Step-by-step instructions, exact tools, exact data to capture.

### Tester pre-requisites

- Logged into Table Salt as an owner-role user on a test tenant.
- Browser: Chrome or Edge with React DevTools and TanStack Query DevTools extensions installed (or use the in-page TanStack Query DevTools panel if enabled).
- Two browser tabs available: tab A and tab B, both signed in as the same user.

### Recon Step 1 — confirm the symptom (rule out H3)

1. Tab A: navigate to Settings → General tab.
2. Open DOM Inspector on the Currency dropdown trigger (`[data-testid="select-currency"]`).
3. Inspect the inner text content of the trigger. Capture: is the text empty, or does it contain "$ US Dollar (USD)" or similar?
4. Open React DevTools → Components → find `SettingsPage` → inspect state. Capture: value of `currency`. Is it `""`, `undefined`, `"USD"`, or `"KRW"`?

**Branch decision:** if trigger contains visible USD text and `currency` state is `"USD"`, **H3 is confirmed** — escalate to user/testers as "not a bug, UX clarity issue." Skip remaining steps.

If trigger is genuinely empty or `currency` state is `""`/`undefined`, **H3 is ruled out** — proceed.

### Recon Step 2 — cold load test (H1 confirmation)

1. Sign out, then sign back in to clear the React Query cache.
2. Immediately navigate to Settings → General tab.
3. Watch the Currency dropdown for a flicker. If hard to catch, screen-record the first 2 seconds.
4. Capture: does the dropdown render with one value (e.g., "USD") and then flicker to another (e.g., "KRW")? Or is it stable from first paint?
5. In Network panel: capture the `GET /api/tenant` request body. Confirm response contains `currency: "KRW"` (or whatever was saved).

**Branch decision:** if flicker observed AND response is correct, **H1 is confirmed**. Continue to Step 3 to check if H2 also applies.

If no flicker AND dropdown shows wrong value stably, **H1 is ruled out** — proceed to Step 3.

### Recon Step 3 — stale cache test (H2 confirmation)

1. Tab A: open Settings, save Currency = KRW. Confirm the save toast appears.
2. Tab A: navigate away (e.g., to Dashboard), then back to Settings. Check Currency value.
3. Tab B: open the same Settings page. Check Currency value.
4. Tab B Network panel: was a `GET /api/tenant` request made when Settings opened? If yes, what was the response? If no, that confirms cache hit.
5. Tab B: hard refresh (Ctrl+Shift+R). Check Currency value now.

**Branch decision:** if tab B shows wrong value AND there was no network request on Settings open, **H2 is confirmed**. If tab B shows correct value or there was a network request, **H2 is ruled out**.

### Recon Step 4 — cache shape inspection (H4 confirmation)

1. Open TanStack Query DevTools panel.
2. Find the cache entry for query key `["/api/tenant"]`. Capture the full data shape — paste into recon doc.
3. Find the cache entry for query key `["/api/tenant", "<tenantId>"]`. Capture its shape too.
4. Compare: are they identical (suggests one is bleeding into the other), or do they have distinctly different field sets (suggests they are separate)?
5. In React DevTools, find `SettingsPage` → inspect the `tenant` value from the useQuery destructure. Compare to the cache entries from steps 2-3.

**Branch decision:** if the Settings page's `tenant` value matches the auth-context's normalized shape (has `subscriptionStatus`, `defaultLanguage`, etc.), **H4 is confirmed**. If it matches only the raw GET shape, **H4 is ruled out**.

### Recon Step 5 — write up findings

1. Tester sends recon results to the team via the standard channel.
2. Each step's outcome is documented as "confirmed" / "ruled out" / "inconclusive — see notes."
3. Path B returns to Claude with the recon results, and Claude proceeds to implement the matching fix candidate from this prep doc.
