## 2. Login Flow

**Files:** `server/routers/auth.ts:126-310`, `server/auth.ts:158-184`

### Passport-Local Strategy (auth.ts:158-184)

1. Check `isAccountLocked(username)` — in-memory map, 5 attempts / 15 min
2. Look up user by username
3. If not found: `recordFailedLogin`, return "Invalid credentials"
4. If `user.active === false`: return "Account is deactivated" — **different message = user enumeration** (F-153)
5. Compare password with scrypt + `timingSafeEqual` — **Good**
6. On success: `clearLoginFailures`, return user

### Post-Authentication (auth.ts → routers/auth.ts:139-310)

1. **TOTP check:** If `totpEnabled && !totpCode`, return `{ requires2FA: true, userId: user.id }` at HTTP 200 — **leaks userId confirming valid creds** (F-154)
2. **Subscription check:** Non-owner roles blocked if subscription expired beyond 24h grace
3. **Session establishment:** `req.login()` — Passport 0.7 regenerates session (fixation protected)
4. **Session data:** `lastActivity`, `idleTimeoutMinutes` (default 30), `sessionToken` (random 36 hex chars)
5. **Concurrent session detection:** Queries other sessions, fires ALERT-13 if found (alert-only, non-blocking)
6. **Session token rotation:** New `randomBytes(18).toString("hex")` stored in DB and session
7. **Session pruning:** Deletes excess sessions beyond `maxConcurrentSessions` (default 5)
8. **Security alerts:** `checkNewIpLoginAlert`, `checkMultiAccountSameIp`, `checkCrossAccountFailedLogins`
9. **Response:** Strips `password`, `totpSecret`, `recoveryCodes`, `passwordHistory`

### Lockout Mechanism (auth.ts:11-41)

- In-memory `Map<string, LockoutEntry>` — **not shared across instances** (F-017)
- 5 attempts / 15 min per username
- Cleared on successful login and on server restart
- **Not per-IP** — attacker can distribute across many usernames without triggering lockout on any

## 3. PIN Flow

**Files:** `server/routers/auth.ts:653-860`, `shared/pin-utils.ts`

### PIN Setting (auth.ts:738)

| Aspect | Implementation |
|--------|---------------|
| Auth | `requireAuth` + `requireFreshSession` |
| Who can set | owner, manager, franchise_owner, hq_admin only |

| Expiry | 90 days from set time |
| Audit | Logged as `pin_set` |

### PIN Login (auth.ts:653)

| Aspect | Implementation | Issue |
|--------|---------------|-------|
| Endpoint | `POST /api/auth/pin-login` | |
| Validation | Must match `/^\d{4}$/` | |
| Lockout | Same in-memory map as password, key `pin:${username}` | Resets on restart |
| Role restriction | Only waiter, cashier, kitchen, delivery_agent, cleaning_staff | Good |
| Comparison | `bcrypt.compare(pin, pinRow.pin_hash)` | Good |
| Expiry check | Enforced at login time | Good |
| Session | Full session created (identical to password login) | |
| **TOTP bypass** | **No TOTP check in PIN login handler** | **2FA bypass** (F-155) |
| **Rate limiting** | **Not covered by authLimiter** — only general 120/min API limiter | **Brute-force risk** (F-145) |
| **CSRF** | **Not in CSRF exemption list** | May break pre-session PIN login (F-143) |

### PIN Brute-Force Analysis

- PIN space: 10,000 combinations (4 digits)
- Weak PINs rejected: ~40 (same-digit: 10, forward seq: 10, reverse seq: 10, user-ID match: ~10)
- Effective space: ~9,960
- At 120 req/min (API limiter): exhausted in **83 minutes**
- App-layer lockout (5 attempts/15 min/username): extends to ~5 hours per target
- But lockout is per-username-per-instance, in-memory, resets on restart

### PIN Change (auth.ts:777)

Requires current PIN + manager credentials (dual authorization). New PIN validated. 90-day expiry reset. Good.

### PIN Reset (auth.ts:835)

Manager/owner can clear PIN (`DELETE /api/auth/pin/:userId`). Same-tenant verified. Audit logged. Good.

### TOTP Setup (auth.ts:427)

| Aspect | Implementation | Issue |
|--------|---------------|-------|
| Auth | `requireAuth` only (no role restriction) | Any user can enable |
| Secret | `otpauth` library `new Secret()` | Good |

| QR code | `QRCode.toDataURL(uri)` | Good |
| **Secret storage** | **Plaintext `text` column** `users.totp_secret` | **Not encrypted** (F-156) |
| Response | Returns secret + QR URL to client | Standard for setup |

### TOTP Verification / Enable (auth.ts:442)

| Aspect | Implementation |
|--------|---------------|
| Validation | `totp.validate({ token: code, window: 1 })` — ±1 period (90s window) |
| Recovery codes | 8 codes, each `randomBytes(4).toString("hex")` (8 hex chars, 32 bits entropy each) |
| **Recovery code storage** | **Plaintext text array** in `users.recovery_codes` | **Not hashed** (F-157) |
| Enable flag | Sets `totpEnabled: true` |

### TOTP at Login (auth.ts:139-161)

- If `totpEnabled && !totpCode`: returns `{ requires2FA: true, userId }` — confirms valid credentials before 2FA
- If code provided: validates with window=1; on failure, checks recovery codes as fallback
- Recovery code consumed on use (spliced from array, array updated in DB) — single-use
- **No TOTP replay protection** within same time window

### TOTP Disable (auth.ts:461)

- Requires current password verification — good
- Sets `totpEnabled: false, totpSecret: null, recoveryCodes: null`
- Audit logged + security alert fired

### TOTP Enforcement

**Entirely optional for all roles.** No mechanism to require 2FA for owner/manager/admin roles. No tenant-level enforcement setting.

## 5. Password Handling

### Hashing (auth.ts:49-53)

```typescript
const salt = randomBytes(16).toString("hex");
const buf = (await scryptAsync(password, salt, 64)) as Buffer;
return `${buf.toString("hex")}.${salt}`;
```

- **Algorithm:** scrypt (not bcrypt, despite bcrypt being a dependency)
- **Output:** 64 bytes
- **Salt:** 16 random bytes per password
- **Comparison:** `timingSafeEqual` — correct
- **Assessment:** Strong. scrypt is memory-hard and appropriate for password hashing.

### Password Policy (auth.ts:62-106)

```
DEFAULT_PASSWORD_POLICY:
  minLength: 8
  requireUppercase: true
  requireLowercase: true
  requireDigit: true
  requireSpecialChar: true
  preventReuseCount: 5
  expirationDays: 90
```

**Where enforced:**
- `POST /api/auth/change-password` — **YES** (auth.ts:490-494)
- `POST /api/auth/reset-password` — **YES** (auth.ts:563)
- `POST /api/auth/register` — **NO** (never called)
- `POST /api/users` (staff creation) — **NO** (default "demo123" assigned)

## 6. Session Security

**File:** `server/auth.ts:122-153`

### Cookie Configuration

| Flag | Value | Assessment |
|------|-------|------------|
| `name` | `ts.sid` | Good (non-default name) |
| `httpOnly` | `true` | Good |

| `sameSite` | `"lax"` | Adequate (prevents most CSRF, allows top-level navigation) |
| `maxAge` | 30 days | Long — appropriate for POS (staff don't want frequent re-logins) |

### Session Store

- **Store:** `connect-pg-simple` (PostgreSQL) — shared across instances
- **Prune interval:** 15 minutes
- **TTL:** 30 days

### Idle Timeout (auth.ts:241-258)

- `requireAuth` checks `session.lastActivity` vs `idleTimeoutMinutes` (default 30, configurable per outlet)
- Expired sessions: `req.logout()` + `req.session.destroy()` — properly cleaned up
- `lastActivity` updated on every authenticated request

### Session Fixation

- Passport 0.7 regenerates session on `req.login()` — protected

### Concurrent Session Handling

- At login: queries for other active sessions, fires alert (non-blocking)
- `requireFreshSession`: compares `session.sessionToken` with `users.session_token` — on mismatch, fires ALERT-13 (alert-only, non-blocking)
- Session pruning: excess sessions beyond `maxConcurrentSessions` (default 5) deleted at login

### Logout (auth.ts → routers/auth.ts:313-322)

| Aspect | Implementation | Issue |
|--------|---------------|-------|
| `req.logout()` | Called | Removes passport data from session |
| `req.session.destroy()` | **NOT called** | Session row persists in DB for 30 days |
| Cookie clear | **NOT explicit** | Browser retains `ts.sid` cookie |
| `sessionToken` rotation | **NOT done** | Other sessions unaffected |

**"Logout All" (`POST /api/auth/sessions/logout-all`):** Rotates `sessionToken`, deletes all other sessions from DB. Correct.

## 7. Encryption Module

**File:** `server/encryption.ts`

| Aspect | Implementation | Assessment |
|--------|---------------|------------|
| Algorithm | AES-256-GCM | Strong AEAD cipher |
| Key derivation | `scryptSync(ENCRYPTION_KEY, SALT, 32)` | Good |
| **Salt** | **Hardcoded: `"table-salt-encryption-v1"`** | **Static salt** — all deployments with same ENCRYPTION_KEY derive the same key |
| IV | `randomBytes(16)` per encryption | Good — unique per field |
| Auth tag | GCM auth tag stored with ciphertext | Good |
| Format | `enc:{iv_hex}:{authTag_hex}:{ciphertext_hex}` | Good — parseable, prefixed |
| Key caching | Derived key cached in module-level variable | Acceptable |
| Error handling | On decrypt failure, returns ciphertext as-is (no crash) | **Permissive** — silently returns encrypted data to caller |

### What Is Encrypted

- **User PII:** `email`, `phone` (via `encryptPiiFields` in storage.ts on user creation/update)
- **NOT encrypted:** `users.totp_secret`, `users.recovery_codes`, `tenants.razorpayKeyId`, `tenants.razorpayKeySecret`

### Q-015 Answered

The encryption module uses AES-256-GCM with scrypt key derivation, random IVs, and authenticated encryption. The static salt is a minor concern (reduces key diversity across deployments sharing the same ENCRYPTION_KEY), but the primary risk remains F-001 (ENCRYPTION_KEY committed to repo in `.replit`).

### AccessMode Bypass (F-095 Verified)

The `accessMode` value comes from `req.body` at line 139. The READ_ONLY check at line 367 is:
```
if (impersonation.access_mode === "READ_ONLY" && mutatingMethods.has(method)) { block }
```

Any value other than the exact string `"READ_ONLY"` (e.g., `"WRITE"`, `"rw"`, `""`, `null`) passes the check and allows full write access. The field is not validated against an enum of allowed values.

## 10. CSRF Protection

**File:** `server/security.ts:181-221`

### Implementation

- **Token computation:** `HMAC-SHA256(sessionId, SESSION_SECRET)` — deterministic per session
- **Cookie:** `csrf-token`, `httpOnly: false`, `sameSite: strict`, `secure` in production
- **Validation:** `x-csrf-token` header must match the cookie value
- **Scope:** All POST/PUT/PATCH/DELETE to `/api/*` (except exempted paths)

### CSRF Exemption Assessment

| Path | Reason | Justified? |
|------|--------|-----------|
| `/api/auth/login` | Pre-session — no CSRF cookie exists yet | **Yes** |
| `/api/auth/register` | Pre-session | **Yes** |
| `/api/auth/forgot-password` | Pre-session, no state change beyond token creation | **Yes** |
| `/api/auth/reset-password` | Pre-session, token-authenticated | **Yes** |
| `/api/webhooks/stripe` | Server-to-server webhook, Stripe-signature authenticated | **Yes** |
| `/api/guest/*` | Guest QR flow — no session, token-authenticated | **Yes** |
| `/api/kiosk/*` | Kiosk flow — bearer-token authenticated | **Yes** |
| `/api/table-requests` | Guest QR submission — no session | **Yes** |
| `/api/errors/client` | Client error logging — no state change | **Yes** |
| `/api/aggregator/webhook/*` | Server-to-server webhook | **Conditional** — needs its own signature verification |
| `/api/ad-impressions` | Unauthenticated impression tracking | **Risky** — allows cross-site impression inflation |

### Missing Exemption

`POST /api/auth/pin-login` is NOT exempted but is a pre-session endpoint. Users logging in via PIN may not have a valid CSRF cookie. This could break the PIN login flow (F-143).

### Missing from Webhook Exemptions

`POST /api/webhooks/razorpay` is NOT in the CSRF exemption list at `security.ts:198-209`. However, the Razorpay webhook handler is registered BEFORE the CSRF middleware (in `index.ts:65`, before `registerRoutes` which calls `setupCsrf`). So CSRF is not applied. This is correct by accident of registration order.

## 11. Rate Limiting

**File:** `server/security.ts:118-173`

### Limiters

| Limiter | Window | Limit | Key | Endpoints | Store |
|---------|--------|-------|-----|-----------|-------|
| `authLimiter` | 15 min | 15 | IP | `/api/auth/login`, `/api/auth/register` | Redis or in-memory |
| `apiLimiter` | 60 sec | 120 | User ID (or IP) | All `/api/` (skips login/register) | Redis or in-memory |
| `uploadLimiter` | 60 sec | 10 | User ID (or IP) | `/api/upload` | Redis or in-memory |

### Coverage Gaps

| Endpoint | Should Have | Has | Risk |
|----------|-----------|-----|------|
| `POST /api/auth/pin-login` | authLimiter (15/15min) | apiLimiter (120/min) | 4-digit PIN brute-force in ~83 min |
| `POST /api/auth/forgot-password` | authLimiter | apiLimiter (120/min) | Email flooding |
| `POST /api/auth/2fa/verify` | authLimiter | apiLimiter | TOTP brute-force (limited by 30s window) |
| `POST /api/auth/change-password` | authLimiter | apiLimiter | Password spraying |
| Admin endpoints | Elevated limiter | apiLimiter only | Mass data exfiltration |

### In-Memory Fallback

When `REDIS_URL` is not set, all rate limiters use in-memory stores. In a multi-instance deployment, each instance tracks limits independently — an attacker can multiply their effective rate by the number of instances.

## Summary of Positive Security Practices

1. **Password hashing:** scrypt with 64-byte output, 16-byte salt, timing-safe comparison
2. **PIN hashing:** bcrypt cost factor 10 with weak-PIN rejection
3. **Reset token:** 256-bit random, SHA-256 hashed in DB, single-use, 1-hour expiry
4. **Session fixation:** Passport 0.7 auto-regenerates session
5. **CSRF:** HMAC double-submit cookie with strict SameSite
6. **Cookie flags:** httpOnly, secure (prod), sameSite lax
7. **PII encryption:** AES-256-GCM with random IVs for email/phone
8. **Concurrent session alerting:** ALERT-13 on mismatch, session pruning
9. **Password change:** Policy + history + audit
10. **Reset:** Destroys all sessions, enforces policy
11. **Impersonation:** Read-only default, dual-step edit unlock, tenant opt-out, full audit trail
12. **Account enumeration:** Protected on forgot-password (same response for all)
