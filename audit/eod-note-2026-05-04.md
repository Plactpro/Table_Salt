## 6 — Diagnostic lessons captured today

**Phase 1 hypotheses can be wrong; later phases are allowed to overrule earlier ones.** F-234 Phase 1 proposed a 3-status predicate set; Phase 3 corrected it to a 9-status NOT IN form after seeing the actual production status distribution. F-247 Phase 1 said "the 404 and the qrDataUrl crash are causally linked"; Phase 2 production data refuted the link for the active failure (today's bills exist). The pattern works: Phase 1 reads code, Phase 2 reads data, Phase 3 reconciles. Each phase is allowed — and sometimes obligated — to update its predecessors. Locking decisions too early before Phase 2 data arrives is a real failure mode.

**Confident wrong citations are a real failure mode.** F-234 Phase 3's first draft cited a non-existent file (`audit/regression-test-plan-step1.md`) at a specific line number to support a claim about test infrastructure. Caught by the operator on spot-check. Replacing a fake citation with another fake citation is also a failure mode and was attempted on the first correction. Lesson: line-number citations are easy to fabricate and hard to spot-check. Verify file existence and line range bounds before approving a memo. If a claim cannot survive citation removal, remove the claim, not just the citation.

**The "show me, then I confirm" gate works.** Today produced 9 commits, 5+ pushes, one production data write (the 67-row F-234 cleanup), and one type-clean code edit. Zero rollbacks needed. Two unintended actions occurred — the rejected backlog edit during the F-234 push window, and the push-without-gate on F-247 Phase 3 after a terse "push it" — both were caught and corrected without harm. The gate caught what the gate is supposed to catch. The gate cost on the order of 30 seconds per commit; without it the rollback cost on either of those two events would have been measurable in tens of minutes.

**PowerShell + tsc + `2>&1` has a quirk.** Native executable stderr redirection on PowerShell 5.1 wraps each line as a `NativeCommandError` ErrorRecord and sets `$?` to false even when the executable returned exit code 0, breaking pipeline-style filters like `... 2>&1 | Select-String "BillPreviewModal"`. Workaround: `tsc` writes errors to stdout, so `2>&1` is unnecessary in the first place — drop it, or capture to a variable (`$out = npx tsc ...`), or pipe via Bash subshell. Worth knowing for future TypeScript verification scripts; this is the kind of platform quirk that eats a 5-minute time slice for no good reason.
